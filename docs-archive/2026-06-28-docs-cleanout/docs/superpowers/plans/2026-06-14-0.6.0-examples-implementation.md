# Part I — Design

## I.1 Current Model

- **Fact 1:** The existing `multi_source_queue` / `multi_flow` examples demonstrate *static* multi-source fan-in but neither concurrent scheduling nor multi-worker. [spec §Purpose lines 22–24]
- **Fact 2:** `batch_stats` is batch-aware (`is_batch_aware=True`): the engine buffers rows and calls `process()` with a `list[PipelineRow]` only when the trigger fires. Its option schema **requires** `value_field: str`, accepts `group_by: str|None` (default None) and `compute_mean: bool` (default True), plus a `schema:` config block with `mode: observed`. [recon R1; `src/elspeth/plugins/transforms/batch_stats.py:30-48, 129, 158`]
- **Fact 3:** `batch_stats` always emits `count`, `sum`, `batch_size`; `mean` is conditional on `compute_mean=True`; `group_by` field appears only if configured; `skipped_*` fields appear only when those conditions occur. Its output contract overrides to `mode=observed`, `fields=None` so input declarations never leak to output. [recon R1; `batch_stats.py:168-190, 222-252, 418-427`]
- **Fact 4:** Two named CSV sources can publish to the same queue via `on_success: <queue_name>`; an explicit `queues:` node makes multi-source fan-in legal and race-free, and the downstream node reads it via `input: <queue_name>`. [recon R1; verified `examples/multi_source_queue/settings.yaml` (both sources `on_success: inbound`, explicit `queues: {inbound: {}}`, transform `input: inbound`), `examples/batch_aggregation/settings.yaml`]
- **Fact 5:** `elspeth join <run_id>` attaches a follower to a RUNNING run. Signature (verified live via `elspeth join --help`): `run_id` positional [required], `--settings/-s` (defaults to `settings.yaml` in cwd; **required in practice** — settings are needed to validate config-hash compatibility, a missing/divergent file is an admission refusal), `--database/-d` (defaults to `settings.landscape.url`), `--format/-f`. **There is NO `--execute` flag on `join`** — join executes unconditionally; there is no dry-run/safety toggle. Exit codes (from help + cli.py:2287-2438): `0` clean departure, `1` JoinRefusedError, `2` FollowerSeatDeadError (resume required), `3` SIGINT/RunWorkerEvictedError, `4` Tier-1/unknown. [recon R2; verified `elspeth join --help`, `cli.py:2287-2310`]
- **Fact 6:** The follower computes `stable_hash(resolve_config(settings))` and admission refuses (JoinRefusedError) unless it equals `runs.config_hash`; therefore the follower MUST pass the identical settings file as the leader. The join window is: `runs.status='RUNNING'` AND `leader_heartbeat_expires_at > now`, checked atomically in one `BEGIN IMMEDIATE` transaction. [recon R2; `core.py:1378-1388`, `run_coordination_repository.py:781-818`]
- **Fact 7:** Audit DB schema (epoch 21, `src/elspeth/core/landscape/schema.py`): `runs.run_id` (PK), `runs.status` (`'running'` for RUNNING — lowercase per `RunStatus.RUNNING`); `token_work_items.run_id`, `token_work_items.status` (`'leased'`, `'terminal'`, `'failed'` per `TokenWorkStatus`), `token_work_items.lease_owner` (worker identity); `run_workers.worker_id` (PK), `.run_id`, `.role` (`'leader'|'follower'`), `.status` (`'active'|'departed'|'evicted'`). **Source attribution lives on `run_sources`, NOT on `rows`:** `run_sources` has `(run_id, source_node_id, source_name)` with PK `(run_id, source_node_id)`; `rows` has `source_node_id` and `source_row_index` but **no `source_name` column**. [recon R3; verified `schema.py:200-223` (run_sources), `schema.py:318-335` (rows — `source_node_id` at :323, no `source_name`), `contracts/scheduler.py:12-20`, `contracts/enums.py RunStatus`]
- **Fact 8:** House run.sh conventions: `set -euo pipefail`, `SCRIPT_DIR`/`PROJECT_ROOT` derivation, `cd "$PROJECT_ROOT"`, `cleanup()` + `trap cleanup EXIT`, ChaosLLM started with `--workers 1` (documented errorworks bug), `/health` curl poll (1–30 iters × 0.5s), `rm -f` of `runs/audit.db{,-wal,-shm}` + output sinks before run. `output/` and `runs/` are git-ignored globally via `examples/**/output/*` + `examples/**/runs/*`; ChaosLLM examples carry **no** `.gitkeep` files. [recon R4; `examples/chaosllm_sentiment/run.sh`, `.gitignore:211-217`]
- **Fact 9:** The `elspeth run` CLI pre-flight (`cli.py:178-182`) creates the landscape DB parent directory with `parent_dir.mkdir(parents=True, exist_ok=True)` before opening the database. So a missing `runs/` directory at clone time does NOT fail a run — the directory is auto-created. The only consequence of an absent committed dir is that a clean clone lacks the empty `runs/`/`output/` directories (cosmetic). [verified `cli.py:178-182`; resolves the D6 rationale — it is the CLI pre-flight, NOT "SQLite auto-creates"]

## I.2 End-State Decisions

### D1 — `concurrent_scheduler` uses the recon-R1 verbatim count-6 two-source settings
- **Decision:** Two CSV sources (`stream_a`, `stream_b`, 3 rows each) → shared queue `merged` → one `batch_stats` aggregation `trigger: {count: 6}`, `output_mode: transform` → CSV sink. `value_field: amount`, `compute_mean: true`, **no `group_by`** (single rendezvous row over all 6 members — spec lines 96–98, recon R1 OPEN RISKS).
- *Integrity argument:* A count-6 barrier can only fire if all six tokens are alive concurrently. One-at-a-time draining would deadlock the barrier (it never accumulates 6 members), so successful completion *is* the proof of concurrent scheduling. This renders the C2 rendezvous proof runnable. [spec lines 43–48]
- *Resolves spec open-question #2:* `batch_stats` exact options finalised — `value_field: amount`, `compute_mean: true`, `group_by` omitted. [spec lines 197–199; recon R1]
- *Sink schema = `mode: observed`* (not fixed) so the variable `batch_stats` output field set (`count`/`sum`/`batch_size`/`mean` plus any conditional `skipped_*`) never trips fixed-field validation. [recon R1 OPEN RISKS]
- *No `collision_policy` on the sink:* the sink writes `rendezvous.csv` and relies on the run.sh-absent `rm -f` in Task 1.3 Step 1 plus the Phase 1 gate to start clean (matching `batch_aggregation`, which uses default overwrite). A `collision_policy: auto_increment` would write `rendezvous_1.csv` on an un-cleaned re-run and defeat the `cat output/rendezvous.csv` verification, so it is deliberately omitted. [W3]

### D2 — `multi_worker` is ChaosLLM-backed, ~60 rows, leader+1-follower default
- **Decision:** Model `settings.yaml` on `chaosllm_sentiment` (an `llm` transform → `base_url: http://127.0.0.1:8199/v1`, `model: chaosllm/fake-gpt-4`, fake `api_key`), input **60 rows**, with a `retry:` block and a `quarantine` sink. `run.sh` backgrounds the leader, polls the audit DB for RUNNING + ≥1 `leased` work item, launches `WORKERS` (env, default `1`) followers via `elspeth join <run_id>` (no `--execute`), reaps all, and asserts **≥2 distinct `lease_owner`s each completed ≥1 row** → prints `✓ PASS` / `✗ FAIL`, non-zero exit on FAIL. [spec §2 lines 100–137]
- *Integrity argument:* Per-worker attribution from `token_work_items.lease_owner` joined to `run_workers.role` is the audit-true evidence that the leader and follower(s) actually shared the workload (recon R3 Query 2). Read-only DB access (`mode=ro` / `PRAGMA query_only`) means the verification queries never contend with live worker writes (resolves spec open-question #3).
- *Resolves spec open-questions #1, #3, #4:* join-window timing (D4 below), read-only DB access (recon R3), and exact column names (recon R3 Fact 7).

### D3 — `multi_worker_showcase` scales D2 to 4 workers / ~200 rows, demonstrative-only
- **Decision:** Multiple named sources fanning in + ~200 rows + a 4-worker pack (1 leader + `WORKERS` default **3** followers) against ChaosLLM. `run.sh` reuses the D2 leader→poll→join→reap skeleton (copied & commented, not shared — suite convention), tails combined live progress, and on completion prints an ASCII "stats card" (workers spawned, total rows, aggregate rows/sec, succeeded/quarantined). **No assertions, no PASS/FAIL.** Exit 0 + stats card rendered is the only bar. [spec §3 lines 139–155]
- *Integrity argument:* Explicitly framed in its README as "a demo to watch, not a proof"; it points readers to `multi_worker/` and `concurrent_scheduler/` for the rigorous checks. No correctness claim is made, so no assertion is owed.
- *CI-cost fence:* heaviest workload (~200 rows × 4 workers); marked in AGENTS.md as not-for-dogfood-gate, like `chaosllm_endurance` (resolves spec open-question #5). [spec lines 208–210]
- *Multi-producer fan-in requires an explicit queue:* both `feed_a` and `feed_b` publish to the same connection name `source_out` (`on_success: source_out`), an explicit `queues: {source_out: {}}` node makes the multi-producer fan-in legal, and the `llm` transform reads `input: source_out`. Without the explicit queue node the DAG validator rejects multi-producer fan-in at startup and the run never begins. (Mirrors the `multi_source_queue` named-queue pattern.) [B3; verified graph.py:352-358 fan-in rule + `multi_source_queue/settings.yaml`]

### D4 — Join-window timing is handled by input sizing + a bounded readiness poll that requires a claimed work item
- **Decision:** The `run.sh` poll loop (recon R2/R3/R5) does **not** approve a join on `status='running'` alone — it additionally requires `≥1 token_work_items` row with `status='leased'` for that `run_id`, proving the leader has begun processing before any follower attaches. Input is sized to **60 rows** (ex2) / **200 rows** (ex3) so that, at ChaosLLM steady-state latency (`base_ms: 30, jitter_ms: 20` plus `slow_response` and injected faults+retries), the leader cannot drain the input before the poll launches at least one follower. **No artificial `sleep` is used to widen the window** — if the race proves tight during empirical validation (Task 2.6 / 3.5), raise the row count rather than add sleeps. [spec open-question #1 lines 189–196; recon R2/R5 OPEN RISKS]
- *Integrity argument:* The readiness criterion (RUNNING ∧ ≥1 leased item) is the strongest signal available pre-join; it prevents a follower joining while the leader is still ingesting (safe-but-inelegant) and gives the assertion the widest honest window. The degenerate "leader finishes first" case is intrinsic to the RUNNING→terminal state machine and cannot be removed in code; it is mitigated empirically and the README documents it.

### D5 — Read-only attribution queries verbatim from recon R3
- **Decision:** Both DB reads use `sqlite3 "file:examples/<name>/runs/audit.db?mode=ro"` and `PRAGMA query_only = ON;`. Query 1 (discovery): `SELECT run_id FROM runs WHERE status='running' LIMIT 1`. Query 2 (attribution): `token_work_items` LEFT JOIN `run_workers` grouped by `lease_owner`, counting `status IN ('terminal','failed')`. Source-attribution proof (concurrent_scheduler, ex1) uses `run_sources` — **not** the non-existent `rows.source_name` column. [recon R3 RECOMMENDED ARTIFACT; B2]
- *Integrity argument:* WAL readers never block writers; the LEFT JOIN is null-safe for recovered/edge-case items where `lease_owner` may be NULL or `run_workers` registration lags.

### D6 — Directory/gitignore convention: no `.gitkeep`, rely on global ignore; CLI pre-flight auto-creates the dir
- **Decision:** Create `output/` and `runs/` directories per example but add **no** `.gitkeep` files — matching the existing ChaosLLM examples (recon R4 Fact: ChaosLLM examples have no `.gitkeep`). `.gitignore` already covers `examples/**/output/*` and `examples/**/runs/*` globally; **no new `.gitignore` entries are required** (verified recon R4 `.gitignore:211-217`). For `concurrent_scheduler` (no `run.sh` to recreate `runs/`), the directory does **not** need to exist at clone time: the `elspeth run` CLI pre-flight (`cli.py:178-182`) does `parent_dir.mkdir(parents=True, exist_ok=True)` and auto-creates it (Fact 9), so the run does not fail on a clean clone — the only effect is the empty dir being absent from a clone (cosmetic). Optionally add a local `.gitignore` (the `chaosllm_endurance` pattern) or a `.gitkeep` (the `multi_source_queue`/`multi_flow` pattern) purely for tidiness; it is not load-bearing. [W4]

### Task 2.4: Create `run.sh` (leader → DB-poll → join → reap → attribution assertion)

**Files:**
- Create: `examples/multi_worker/run.sh` (then `chmod +x`)

- [ ] **Step 1: Write `run.sh`.** Combine the recon-R4 house skeleton (set -euo pipefail, SCRIPT_DIR/PROJECT_ROOT, ChaosLLM start + `/health` poll, `trap cleanup EXIT`) with the recon-R2 leader+follower choreography and the recon-R3 read-only SQL (verbatim). The `cleanup()` must kill ChaosLLM **and** any still-running worker PIDs (spec lines 136–137). The poll loop implements D4 (require RUNNING ∧ ≥1 `leased` item). Status literals are **lowercase** (`'running'`, `'leased'`, `'terminal'`, `'failed'`) per Fact 7. **Followers are launched with NO `--execute` flag** (B1 — `join` has no such flag). The `cleanup()` follower loop is array-length-guarded so an empty `FOLLOWER_PIDS` under `set -u` does not expand to a single empty element (W2):
```bash
#!/usr/bin/env bash
# =============================================================================
# multi_worker — elspeth join: independent processes share one RUNNING run
#
# Backgrounds a LEADER (elspeth run), polls the audit DB read-only until the run
# is RUNNING with >=1 claimed work item, then launches WORKERS (default 1)
# FOLLOWERS via `elspeth join <run_id>`. After all processes exit it queries
# token_work_items grouped by lease_owner and asserts >=2 distinct workers each
# completed >=1 row.  ADR-030 "One-Host WAL Pack".
#
# Usage:   ./examples/multi_worker/run.sh        # leader + 1 follower (2-way)
#          WORKERS=3 ./examples/multi_worker/run.sh   # leader + 3 followers
# =============================================================================
set -euo pipefail

SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
PROJECT_ROOT="$(cd "$SCRIPT_DIR/../.." && pwd)"
cd "$PROJECT_ROOT"

CHAOS_CONFIG="examples/multi_worker/chaos_config.yaml"
PIPELINE_CONFIG="examples/multi_worker/settings.yaml"
DB="examples/multi_worker/runs/audit.db"
CHAOS_PORT=8199
CHAOS_PID=""
LEADER_PID=""
FOLLOWER_PIDS=()
WORKERS="${WORKERS:-1}"

cleanup() {
    # Kill any still-running followers and the leader, then the ChaosLLM server.
    # Array-length-guarded so an empty FOLLOWER_PIDS under `set -u` does not
    # expand to one empty-string element (which would run `kill -0 ""`).
    if [ ${#FOLLOWER_PIDS[@]} -gt 0 ]; then
        for pid in "${FOLLOWER_PIDS[@]}"; do
            kill -0 "$pid" 2>/dev/null && kill "$pid" 2>/dev/null || true
        done
    fi
    [ -n "$LEADER_PID" ] && kill -0 "$LEADER_PID" 2>/dev/null && kill "$LEADER_PID" 2>/dev/null || true
    if [ -n "$CHAOS_PID" ] && kill -0 "$CHAOS_PID" 2>/dev/null; then
        echo ""
        echo "Stopping ChaosLLM server (PID $CHAOS_PID)..."
        kill "$CHAOS_PID" 2>/dev/null || true
        wait "$CHAOS_PID" 2>/dev/null || true
    fi
}
trap cleanup EXIT

# Clean previous run artifacts
rm -f examples/multi_worker/runs/audit.db examples/multi_worker/runs/audit.db-wal examples/multi_worker/runs/audit.db-shm
rm -f examples/multi_worker/output/results.json examples/multi_worker/output/quarantined.json
mkdir -p examples/multi_worker/runs examples/multi_worker/output

echo "=== multi_worker (elspeth join) — leader + $WORKERS follower(s) ==="
echo ""

# --- Start ChaosLLM (errorworks bug: must use --workers 1) ---
echo "Starting ChaosLLM server on port $CHAOS_PORT..."
.venv/bin/chaosllm serve --config "$CHAOS_CONFIG" --port "$CHAOS_PORT" --workers 1 &
CHAOS_PID=$!
echo "Waiting for ChaosLLM to be ready..."
for i in $(seq 1 30); do
    if curl -sf "http://127.0.0.1:$CHAOS_PORT/health" > /dev/null 2>&1; then
        echo "ChaosLLM is ready."; echo ""; break
    fi
    if ! kill -0 "$CHAOS_PID" 2>/dev/null; then echo "ERROR: ChaosLLM failed to start."; exit 1; fi
    sleep 0.5
done
if ! curl -sf "http://127.0.0.1:$CHAOS_PORT/health" > /dev/null 2>&1; then
    echo "ERROR: ChaosLLM not responding after 15 seconds."; exit 1
fi

# --- Launch LEADER (background) ---
echo "Launching leader: elspeth run --execute ..."
.venv/bin/elspeth run --settings "$PIPELINE_CONFIG" --execute &
LEADER_PID=$!

# --- Poll audit DB (read-only) for RUNNING run with >=1 claimed work item ---
# Readiness criterion (design D4): RUNNING is not enough; require >=1 'leased'
# token_work_item so the leader has demonstrably begun processing before a
# follower attaches. Bounded retry; no artificial sleep widens the join window.
RUN_ID=""
for attempt in $(seq 1 60); do
    RUN_ID="$(sqlite3 "file:${DB}?mode=ro" \
        "PRAGMA query_only=ON; SELECT run_id FROM runs WHERE status='running' LIMIT 1;" 2>/dev/null || true)"
    if [ -n "$RUN_ID" ]; then
        CLAIMED="$(sqlite3 "file:${DB}?mode=ro" \
            "PRAGMA query_only=ON; SELECT COUNT(*) FROM token_work_items WHERE run_id='$RUN_ID' AND status='leased';" 2>/dev/null || echo 0)"
        if [ "${CLAIMED:-0}" -ge 1 ]; then
            echo "Leader RUNNING (run_id=$RUN_ID) with $CLAIMED claimed item(s); joining."
            break
        fi
    fi
    # Guard: leader may have already finished (degenerate fast-drain race, design risk #1)
    if ! kill -0 "$LEADER_PID" 2>/dev/null; then
        echo "WARNING: leader exited before a follower could join (fast-drain race)." >&2
        break
    fi
    sleep 0.5
done
if [ -z "$RUN_ID" ]; then
    echo "FAIL: leader never reached RUNNING within the poll window." >&2
    exit 1
fi

# --- Launch FOLLOWERS (same --settings => identical config_hash; NO --execute) ---
for i in $(seq 1 "$WORKERS"); do
    echo "Launching follower $i: elspeth join $RUN_ID ..."
    .venv/bin/elspeth join "$RUN_ID" --settings "$PIPELINE_CONFIG" &
    FOLLOWER_PIDS+=("$!")
done

# --- Reap leader + followers ---
wait "$LEADER_PID"; LEADER_EXIT=$?
echo "Leader exited ($LEADER_EXIT)."
for pid in "${FOLLOWER_PIDS[@]}"; do
    wait "$pid" || echo "Follower $pid exited non-zero ($?) (see exit-code semantics in README)."
done

# --- Attribution query (read-only): per-worker completed rows ---
echo ""
echo "Per-worker attribution (token_work_items grouped by lease_owner):"
sqlite3 "file:${DB}?mode=ro" <<SQL
PRAGMA query_only = ON;
SELECT
  t.lease_owner,
  w.role,
  COUNT(CASE WHEN t.status IN ('terminal', 'failed') THEN 1 END) AS completed_rows
FROM token_work_items t
LEFT JOIN run_workers w ON w.worker_id = t.lease_owner AND w.run_id = t.run_id
WHERE t.run_id = '$RUN_ID'
GROUP BY t.lease_owner, w.role
ORDER BY w.role DESC, completed_rows DESC;
SQL

# --- Assertion: >=2 distinct workers each completed >=1 row ---
WORKER_COUNT="$(sqlite3 "file:${DB}?mode=ro" \
  "PRAGMA query_only=ON; SELECT COUNT(*) FROM (SELECT lease_owner FROM token_work_items WHERE run_id='$RUN_ID' AND status IN ('terminal','failed') AND lease_owner IS NOT NULL GROUP BY lease_owner HAVING COUNT(*) >= 1);" 2>/dev/null || echo 0)"
TOTAL_ROWS="$(sqlite3 "file:${DB}?mode=ro" \
  "PRAGMA query_only=ON; SELECT COUNT(*) FROM token_work_items WHERE run_id='$RUN_ID' AND status IN ('terminal','failed');" 2>/dev/null || echo 0)"

echo ""
if [ "${WORKER_COUNT:-0}" -ge 2 ]; then
    echo "✓ PASS: leader + $((WORKER_COUNT-1)) follower(s) shared $TOTAL_ROWS rows across $WORKER_COUNT workers"
    exit 0
else
    echo "✗ FAIL: only ${WORKER_COUNT:-0} worker(s) completed rows; expected >=2 (join-window race? raise WORKERS or row count)" >&2
    exit 1
fi
```
