# ELSPETH Platform Architecture

**Document date:** 5 June 2026
**Release covered:** RC-5.3
**Audience:** Evaluators, technical leaders, architects, and assurance reviewers
**Register:** Public-facing / technical
**Status:** Current architecture overview

## Summary

ELSPETH is built around one architectural rule: an output is not complete until
the system can explain how it was produced.

The platform separates authoring, execution, audit evidence, external calls, and
operator-facing review surfaces. That separation lets a user build a pipeline in
the web Composer, run it through the engine, and later ask why any particular
output exists.

## Main Surfaces

| Surface | Role |
|---|---|
| CLI | Runs, validates, resumes, and inspects pipelines from the command line. |
| Web app | Provides authentication, sessions, Composer, execution controls, run progress, and review surfaces. |
| Composer | Builds pipeline configuration through guided or freeform authoring. |
| Engine | Executes the directed pipeline graph and records terminal outcomes. |
| Plugin system | Provides sources, transforms, sinks, and integrations. |
| Landscape audit store | Persists run, row, node, call, payload, secret-resolution, and lineage evidence. |
| Telemetry | Provides operational visibility after audit writes have succeeded. |

## Pipeline Flow

```text
External source
    |
    v
Source plugin
    |
    v
Validated pipeline row
    |
    v
Transform / gate / aggregation graph
    |
    +--> External calls where configured
    |
    v
Sink or routed terminal outcome
    |
    v
Landscape audit evidence
```

The same engine executes YAML written by an operator and YAML generated by
Composer. Composer is an authoring path, not a separate runtime.

## Trust Boundaries

ELSPETH uses a three-tier trust model.

| Tier | What it covers | Behaviour |
|---|---|---|
| Tier 1 | Data ELSPETH authored, such as audit records and checkpoints | Internal inconsistency is treated as a system bug and stops execution. |
| Tier 2 | Validated pipeline data moving between steps | The engine trusts the schema but still wraps operations that can fail on user data values. |
| Tier 3 | External input, including source rows and external API responses | The boundary validates or quarantines data before it can become pipeline data. |

This model is deliberately stricter than conventional "log and continue"
behaviour. For audit-relevant systems, silent recovery from corrupt internal
state is worse than stopping with a clear cause.

## Audit Comes First

Landscape is the source of truth. Telemetry and logs are secondary.

When an operation has evidentiary value, ELSPETH writes the audit record first.
If that write fails, the operation fails. Telemetry can then report operational
state, but it does not replace the audit trail.

This matters because future reviewers need durable evidence, not just runtime
observability. A dashboard can explain what is happening now; Landscape explains
what happened then.

## Configuration And Validation

ELSPETH keeps user configuration and runtime configuration separate:

```text
settings / YAML -> validated settings -> runtime config -> engine components
```

That mapping prevents a field from being accepted in user configuration but
silently ignored at runtime. Runtime configuration contracts and alignment tests
are part of the release discipline.

Pipeline shape is also validated before execution. Sources declare what fields
they can provide, transforms declare what fields they require, and the graph
builder rejects incompatible shapes before rows start moving.

## External Systems

ELSPETH integrates with external systems through plugins and clients, including:

- CSV and JSON files;
- Azure Blob Storage;
- Microsoft Dataverse;
- ChromaDB;
- Azure OpenAI;
- OpenRouter;
- web-scraping targets, with controls intended to prevent requests to internal
  network locations.

Calls to external systems are treated as trust boundaries. The response is not
trusted until it has been parsed, validated, and recorded according to the
plugin's contract.

## Recovery And Resume

Long-running work can be interrupted. ELSPETH records checkpoints at processing
boundaries so a run can resume without reprocessing completed work where the
pipeline and checkpoint remain compatible.

Composer also has an authoring recovery surface. If a composition fails or is
interrupted, the web UI can show the transcript, redacted tool rows, and state
diffs so the user can decide what to retain.

## Security-Relevant Design Choices

- Secrets are referenced by name and resolved at runtime.
- Secret-resolution events record fingerprints, not raw secret values.
- Payload hashes use canonical JSON so records can be compared deterministically.
- Invalid external rows are quarantined rather than silently dropped.
- Authentication supports local, OpenID Connect, and Microsoft Entra providers.
- Shareable review links use signed capability tokens and read-only inspect
  views.

These choices do not remove the need for agency-specific assessment. They make
the system's behaviour inspectable enough for that assessment to begin.

## Operational Boundaries

The platform still requires deployment choices by the adopting team:

- identity provider and group/role policy;
- network controls and allowed external endpoints;
- data classification and provider suitability;
- audit-database retention;
- backup and recovery process;
- schema migration procedure;
- incident response ownership.

ELSPETH provides the engine and evidence model. It does not decide the agency's
risk appetite or authority-to-operate boundary.
