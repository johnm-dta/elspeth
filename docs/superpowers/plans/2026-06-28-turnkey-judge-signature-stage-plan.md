# Turnkey judge/signature stage — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: use `superpowers:subagent-driven-development` (or `superpowers:executing-plans`) to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Date:** 2026-06-28
**Implements:** `docs/superpowers/specs/2026-06-28-turnkey-judge-signature-stage-design.md` (approved for planning).
**Tracks:** elspeth-281582acc9 (signature tooling), elspeth-b3a3335c9f [O1] (operator-only HMAC custody), elspeth-2b351cd004 (CI key exposure).

**Goal:** Build the *stage → sign-bundle* seam: a key-free **agent MCP surface** (`elspeth-judge`) that assembles an authority-free review **bundle**, and an operator **CLI** (`sign-bundle`, `rekey`) that re-verifies the bundle against the source tree and is the *only* place a signature is minted. Compose the existing `justify` / `diagnose-judge-signatures` / `sign-judge-signatures` / `migrate-judge-scope` / `rotate` primitives; do **not** reinvent them.

**Architecture:** A new authority-free `ReviewBundle` JSON artifact under `.elspeth/staged-reviews/` (already gitignored). The agent stages into it via MCP; the operator fires it via `elspeth-lints sign-bundle <bundle>`. The bundle carries *claims*; `sign-bundle` re-derives every binding from the tree (reusing `diagnose_judge_signatures`) and aborts on any drift mismatch before a single write. New judgments run the **real** judge inside the keyed step (reusing the `_run_justify` ceremony). Re-signs split by `kind`: **drift_repair** (missing/invalid/AST-path/scope/binding drift) **re-judges** through that same `_run_justify` ceremony — i.e. exactly what the existing `sign-judge-signatures` already does — because re-judging is what *prevents laundering a stale verdict over drifted or changed content*; **rotation** mechanically re-binds *non-judge-gated* keys through `apply_plan` (no judge, no verdict; the rotation worklist is built by `stage_scan`/verify feeding **only** non-judge-gated entries into `plan_rotations` via `scan_for_rotations(..., exclude_judge_gated=True)` — `apply_plan` itself has **no** judge-gated guard, so the non-judge-gated guarantee is enforced at survey/verify time, not write time); **stale_delete** removes orphaned entries. Writes go through `atomic_update_text`; the fingerprint baseline is regenerated last.

**Tech stack:** Python (the `elspeth_lints` package lives at `elspeth-lints/src/elspeth_lints`), `argparse` CLI, the `mcp` SDK (mirroring `src/elspeth/mcp/server.py`), `pytest`. No new third-party deps.

## Global Constraints

*Every task's requirements implicitly include this section.*

- **Test runner:** existing elspeth-lints tests live under `tests/unit/elspeth_lints/` and import `from elspeth_lints.core.cli import main` directly (the package is installed editable). Run new tests with `.venv/bin/python -m pytest tests/unit/elspeth_lints/<file> -v`. The lint baseline is **interpreter-version-specific** (Python 3.12.x — the canonical `.venv` currently reports 3.12.3; verify live: `.venv/bin/python --version`); match the canonical `.venv`. (CI runs the `python:3.13-bookworm` container, but the local fingerprint-baseline regen shells the *local* interpreter via `regen_fingerprint_baseline.py`'s `VENV_PYTHON=REPO_ROOT/.venv/bin/python` — so do **not** provision a 3.13 worktree venv expecting it to match the canonical baseline.)
- **The five security invariants (spec §5) are first-class TDD tasks, not comments.** Each has a named test below.
- **[O1] linchpin — the agent never holds the HMAC key.** Every MCP tool fails closed when `ELSPETH_JUDGE_METADATA_HMAC_KEY` is present in its environment. There is **no authoritative MCP path** (resolves the §4.2-vs-§5/§7 wording collision toward fail-closed): MCP `verify_signatures` is *always* shape-only; the authoritative recompute path remains the CLI `diagnose-judge-signatures` / library `diagnose_judge_signatures` (which upgrades via `_verification_mode()`). The report's `verification_mode` field is what satisfies the "labels shape-only vs authoritative" requirement.
- **Staging asserts; firing verifies.** A bundle never serializes a signature, and every `preview.authoritative` is `False`. `sign-bundle`/`rekey` re-derive bindings from the tree; a staged claim that disagrees with ground truth aborts the run.
- **Verify is atomic; execute is per-action — and that is correct, not a gap.** `verify_bundle_against_tree` is the *all-or-nothing* gate: it runs over **every** action and aborts the whole run before a single write if any claim is stale (the "abort before any write" property is a **verify-phase** guarantee). The **execute** phase, by reusing the `_run_sign_judge_signatures` continue-on-failure ceremony (`cli.py:3200-3253`), is **per-action non-transactional**: in a multi-action bundle the real judge first runs *after* the confirm gate, so if action *N* honestly `BLOCK`s, actions `1..N-1` are already written and only action *N* is restored/skipped; the command returns non-zero with a partial application and an "M succeeded / K failed" report. This is **safe** — every committed write was judge-authorized, the operation is idempotent/re-runnable, and the BLOCKed action's prior on-disk state is restored (Task 2.3) — but it is an operator-visible behaviour that must be **stated and tested** (Task 2.4), so a 10-action run that blocks at action 5 has an answerable "what state am I in?". Do **not** try to make execute atomic — that would fight the `_run_justify` reuse; the verify gate is the atomicity boundary by design.
- **Signing never runs in CI.** `sign-bundle` and `rekey` are operator-shell-only commands. CI (`enforce-allowlist-judge-gates.yaml`) keeps *verifying* (C3 override-rate, VAL judge-quality); it never signs.
- **Reuse, do not reimplement.** New-judgment signing reuses the `_run_justify` ceremony (which already returns 1 and does not write on `BLOCKED`-without-override). **`drift_repair`** re-sign reuses `_signing_specs_from_diagnosis` → `_namespace_for_signing_spec` → `_run_justify` *verbatim* — and this **re-judges**: `_run_justify` calls `call_judge` unconditionally on its happy path (`cli.py:1699`), and `_signing_specs_from_diagnosis` maps *every* signable status (incl. `AST_PATH_BINDING_DRIFT`) to "one exact `justify` invocation" whose diagnosis `repair_command` is itself `_justify_command`. Writes reuse `atomic_update_text`. Rotations reuse `apply_plan` (which already raises on a dup-key — see the trap below) — **never** add a fresh `str.replace`. `migrate-judge-scope`'s two-pass integrity-gate-first structure is the **structural precedent for `rekey`** (Phase 4), *not* the resign lane.
- **Resign re-judges — this is a security property, not a perf cost.** The spec is internally contradictory: §3's prior-art table classifies `sign-judge-signatures` as "HMAC + **LLM**" (it re-judges), while §4.3 step 3 said `drift_repair` "carry the verdict forward … No judge." §4.3 is wrong about `sign-judge-signatures`. The plan resolves this toward **re-judging** for `drift_repair`, because a no-judge carry-forward over an honest `SCOPE_BINDING_DRIFT` (enclosing-scope content *actually changed*) would stamp a stale ACCEPTED verdict onto code the judge never inspected — the exact [O1] forgery the design forbids. The only genuine no-judge re-sign paths are **`rotation`** (mechanical key re-bind of *non-judge-gated* entries — fed only non-judge-gated entries by `stage_scan`/verify; the judge-gated **refusal fires at SCAN time inside `plan_rotations`** at `rotate.py:399/450` → `_refuse_rotation_of_judge_gated_entry:164`, **not** in `apply_plan`, which has no such guard) and **`stale_delete`** (removal). `migrate_v2` is **dropped** from the schema (a per-entry bundle action cannot reuse the whole-allowlist `migrate-judge-scope` command without reimplementing it; v1→v2 migration stays on the standalone `migrate-judge-scope` CLI). Spec §3/§4.3/§7 are amended to match (Task 5.1 records the spec edit); the plan below is self-consistent and does **not** rely on the spec being read alongside it.
- **Dup-key dataloss trap:** historically `rotate` used a `str.replace` that could delete *both* copies of a duplicated canonical key. The current `apply_plan` path replaces by **YAML-AST node span** and `_replace_allow_hit_key` (`rotate.py:1012`) raises `RuntimeError` when the `old_key` span count is `> 1` ("occurs Nx … refusing") or `0` ("not found") — it never deletes both copies. The plan's job is to route every rotation through `apply_plan`, **catch its bare `RuntimeError` in a narrow `try/except` around the `apply_plan` call** (the narrow scope avoids swallowing unrelated ones) and convert it to a clean `return 2`, and add a test that a dup-key bundle aborts (`== 2`) with *both* copies intact. **Message discipline (do not over-claim):** `apply_plan` raises a bare `RuntimeError` for **at least six distinct reasons** — the genuine dup-key (`old_key` occurs Nx, `rotate.py:1020`), `old_key` not found (span 0, `:1016`), TODO-stub debt (`:568`), file-not-found during apply (`:606`), `allow_hits` not a sequence (`:1040`), and `allow_hits` entry not a mapping (`:1051`). The `return 2` message must **surface the actual exception text** (e.g. `f"rotation apply failed: {exc}"`); reserve the specific **"both copies preserved"** guarantee for the verified `occurs Nx` (`:1020`) case only — the other five are not dup-key and would be mislabeled by a hard-coded dup-key string. **Multi-file caveat:** `apply_plan` writes file-by-file under `atomic_update_text` (`rotate.py:586-621`), so a `RuntimeError` on file *N* leaves files `1..N-1` **already written** — consistent with the "execute is per-action non-transactional" framing; the abort is re-runnable, *not* a clean no-write rollback for a multi-file rotation bundle. Do not word the dup-key message to imply a whole-bundle rollback.
- **AST-position cascade:** tier-model fingerprints are AST-position sensitive — adding an `import` to a covered `src/elspeth` source shifts `Module.body` indices and cascades every downstream finding's fingerprint. This is exactly the staleness `sign-bundle` must catch: a bundle staged before such an edit will mismatch the tree and must abort (cheap to fix: re-run `stage_scan`).

## The bundle — concrete typed structure

`core/review_bundle.py`. Dataclasses (frozen, slots), JSON via `dump_bundle`/`load_bundle`.

```python
@dataclass(frozen=True, slots=True)
class ActionPreview:
    verdict: str            # "ACCEPTED" | "BLOCKED" — model preview only
    rationale: str          # preview rationale (discarded for authority at fire time)
    model: str              # served model id of the preview judge
    transport: str          # "claude_agent_sdk" (the non-authoritative agent transport)
    authoritative: bool = False   # INVARIANT: always False; __post_init__ raises if True

@dataclass(frozen=True, slots=True)
class BundleAction:
    lane: str               # "new_judgment" | "resign"
    kind: str               # justify | drift_repair | rotation | stale_delete  (migrate_v2 dropped — v1→v2 stays on the standalone migrate-judge-scope CLI)
    key: str                # canonical allowlist key
    file_path: str | None = None
    symbol: str | None = None
    rule: str | None = None
    fingerprint: str | None = None
    scope_fingerprint: str | None = None
    ast_path: str | None = None
    draft_rationale: str | None = None        # operator/agent-authored "why"
    diagnosis_status: str | None = None        # resign: the status staging asserts (e.g. AST_PATH_BINDING_DRIFT)
    source_file: str | None = None             # resign/stale: owning YAML filename
    preview: ActionPreview | None = None       # new_judgment only; never authoritative

@dataclass(frozen=True, slots=True)
class RekeyPlan:
    # ALL fields are advisory provenance: the operator `rekey` CLI flags + the live tree are
    # authoritative. `rekey` re-derives the judge-gated set from the tree at fire time and asserts
    # the env-var names match the flags (aborts on mismatch); it never drives Pass-1 off `keys`.
    old_key_env: str        # NAME of the env var holding the old key (never the key bytes)
    new_key_env: str        # NAME of the env var holding the new key
    keys: tuple[str, ...]   # canonical keys shape-valid under the active key (preview — Pass-1 re-derives)
    broken_keys: tuple[str, ...]   # entries that do NOT verify — must be repaired first

@dataclass(frozen=True, slots=True)
class ReviewBundle:
    bundle_id: str
    schema_version: int          # 1
    created_at: str              # ISO8601, stamped by the writer (provenance, not authority)
    staged_by: str               # agent/operator label
    root: str                    # e.g. "src/elspeth"
    allowlist_dir: str           # e.g. "config/cicd/enforce_tier_model"
    source_rev: str | None       # git HEAD sha (provenance; re-verified regardless)
    source_dirty: bool
    actions: tuple[BundleAction, ...]
    rekey: RekeyPlan | None = None   # populated only by stage_rekey
```

**Invariant (enforced in code + tested):** the serialized bundle dict contains **no** signature-shaped key (no `judge_metadata_signature`, no `hmac-sha256:` value), and every `preview.authoritative` is `False`. `ActionPreview.__post_init__` raises `ValueError` if `authoritative is True`; `dump_bundle` never emits a signature field. The key bytes never enter a bundle — only the env-var *names* (`old_key_env`/`new_key_env`).
