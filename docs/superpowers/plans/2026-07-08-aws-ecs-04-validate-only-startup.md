# AWS ECS Validate-Only Web Startup Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (- [ ]) syntax for tracking.

**Goal:** In `ELSPETH_WEB__DEPLOYMENT_TARGET=aws-ecs`, `create_app()` validates schema without creating or mutating it, fails closed to `elspeth doctor aws-ecs` on any non-current state, and refuses to start on any failed `validate_aws_ecs_settings` check — local/default mode keeps today's create-if-missing behavior.

**Architecture:** A new `aws_ecs_startup.py` module owns the connection-retry budget and fail-closed gate; `create_app()` calls it instead of unconditionally calling `initialize_session_schema`, and `_finalize_orphaned_landscape_runs`/`_periodic_orphan_cleanup` gain a `create_tables` flag so orphan reconciliation never mutates schema in aws-ecs mode either.

**Tech Stack:** Python 3.13, SQLAlchemy, pytest, `testcontainers.postgres.PostgresContainer`.

**Depends on:** `2026-07-08-aws-ecs-01-*.md` (`WebSettings.deployment_target`, `deployment_contract.py`), `2026-07-08-aws-ecs-02-*.md` (`schema_probe.py`).

**Global Constraints (verbatim):**
- "If the schema is missing, stale, or only partially present, web startup fails closed and points the operator at the doctor command rather than racing the initializer to completion. Local development can keep create-if-missing behavior." (spec L172-176)
- "...the web app refuses to start unless explicit PostgreSQL session and landscape URLs are provided." (spec L477-478)
- "The startup validate path uses a 10-second connection timeout and a bounded retry with backoff, up to 45 seconds total per probe, before failing." (spec, Aurora Serverless v2 Considerations — amended)
- "Validate-only startup budgets per dependency, not per process: each schema probe (session, then landscape) carries an independent 45-second connection-retry budget. With a single shared Aurora cluster — one cold start — startup completes within roughly 60 seconds. The worst case, two independently paused clusters cold-starting serially, is bounded at roughly 90 seconds plus probe time." (spec, Operational Budgets — amended to the per-probe model this plan implements; the ~90s worst case and the ECS `healthCheck.startPeriod` sizing obligation are now spec text, not a plan-local deviation)

**Deviations:**
- ~~Panel review flagged that two independent 45s per-probe retry budgets can total ~90s, busting the spec's original unqualified 60s ceiling.~~ **Resolved by spec amendment, no longer a deviation:** the spec's Operational Budgets and Aurora Serverless v2 sections now state the per-probe model this plan implements — independent 45s budgets per probe, ~60s single-shared-cluster nominal, ~90s two-cold-cluster worst case, and the ECS `healthCheck.startPeriod` sized to the worst case (Terraform-owned, out of this repo). The reasoning that drove the amendment: a shared budget lets a slow-but-healthy first probe starve the second probe's genuinely-healthy retry — a false hard-failure (and ECS restart loop) in exactly the topology the retry budget exists to survive. Per-probe never false-fails a healthy-but-slow deployment; it only takes longer. Plan 05's probe-wiring doc carries the operator-facing startPeriod line.
- Panel review suggested building the retry loop on `tenacity` (already a pinned dependency) instead of hand-rolling it. Kept hand-rolled: the fake-clock `sleep=`/`monotonic=` injectable-parameter contract this budget gate needs (Task 1's own test suite) doesn't map onto `tenacity.Retrying`'s `stop_after_delay`/`wait_exponential` API without added indirection that isn't worth it for one bounded retry loop.
- Panel review noted `_probe_with_connection_budget` retries any `OperationalError` for the full 45s, including permanent misconfiguration (bad password, security-group block) that will never succeed — not just transient Aurora cold-start. Accepted as a known trade-off: SQLAlchemy/psycopg surface both cases identically, and distinguishing them reliably in-process isn't worth the complexity for a startup-time gate; capping retries observed across consecutive ECS task-launch attempts, if this becomes a real operational problem, belongs at the Terraform/ECS layer, out of this plan's scope.
- Panel review (systems perspective, approve-with-nits) found a third schema-mutation site this plan's stated goal ("validates schema without creating or mutating it") doesn't cover and no sibling plan (checked 01/02/03/05/10) owns either: `src/elspeth/web/execution/service.py:1100` constructs `LandscapeDB(connection_string=..., passphrase=...)` (bare `__init__`, which unconditionally calls `_create_tables()`/`_create_additive_indexes()` — it has no `create_tables` parameter at all) on **every pipeline run**; `src/elspeth/web/composer/tutorial_service.py:204` calls `LandscapeDB.from_url(...)` with `create_tables` defaulted `True` on every tutorial-output projection. `_validate_schema()` runs before either mutation, so a STALE schema already fails closed rather than being silently repaired — the residual is a MISSING/PARTIAL schema (or an additive index under version skew) triggering a real `CREATE` against production Aurora from the request path, which would also hard-fail under a least-privilege, DDL-denied ECS task role. Deliberately **not** folded into this plan as a Task 5: both sites sit in the hot per-run execution path (not startup), fixing `execution/service.py:1100` requires switching its bare `LandscapeDB(...)` construction to the `.from_url(..., create_tables=...)` classmethod — a different, riskier code path than this plan's `web/app.py` startup-only scope — and both need their own TDD cycle against `execution/service.py`'s and `tutorial_service.py`'s own test suites. **Owned by plan 11** (`2026-07-08-aws-ecs-11-landscape-write-gate.md`), an executable subplan — not a tracked follow-up — which routes these two sites *plus* the four `auth/audit.py` writer sites (`:174,197,226,246`, same defaulted-`create_tables` exposure) through one `create_tables`-gated factory, and adds a regression guard banning ungated `LandscapeDB` construction under `src/elspeth/web/`. This plan still owns the `app.py` startup + orphan-reconciliation paths; plan 11 owns every other web-layer construction.

### Task 1: Connection-budget retry gate module

**Files:**
- Create: `src/elspeth/web/aws_ecs_startup.py`
- Test: `tests/unit/web/test_aws_ecs_startup.py`

**Interfaces:**
- Consumes: `elspeth.web.deployment_contract.{ContractCheck, validate_aws_ecs_settings}`, `elspeth.web.schema_probe.SchemaState`, `WebSettings`. (`DEPLOYMENT_TARGET_AWS_ECS` is dropped here — nothing in this module's produced symbols references it; Task 2 imports it directly for the `app.py` wiring, where it's actually used. Listing it here would be an unused import under this project's ruff F401 gate.)
- Produces: `AwsEcsSchemaNotReadyError(RuntimeError)`; `_probe_with_connection_budget(engine, probe, *, label, sleep=time.sleep, monotonic=time.monotonic) -> SchemaState`; `enforce_aws_ecs_contract(settings) -> None`.

`_probe_with_connection_budget` calls `probe(engine)`. Only `OperationalError` (Aurora Serverless v2 waking from 0 ACU) triggers a retry; any other exception (e.g. `TypeError` from a signature drift) propagates immediately, unretried. Backoff is capped-and-hold, not cyclical: 1s, 2s, 4s, then 8s for every subsequent retry. Each retry emits `structlog.get_logger().warning("aws_ecs_schema_probe_retry", label=label, attempt=<1-indexed int>, elapsed_seconds=<float>)` so an operator watching CloudWatch during a slow cold start sees retry progress instead of an apparent hang. Retries continue until `monotonic()` exceeds a 45s deadline, then re-raises. Any returned `SchemaState` — even a failing one — returns immediately, never retried: a successful connection answering MISSING/PARTIAL/STALE isn't a connection problem. `sleep`/`monotonic` are injectable for an instant fake-clock test. `_CONNECT_TIMEOUT_SECONDS = 10`/`_CONNECT_RETRY_BUDGET_SECONDS = 45` are module constants (Task 2 imports and reuses `_CONNECT_TIMEOUT_SECONDS` rather than restating the literal); per-attempt timeout is the caller's `connect_args`. `enforce_aws_ecs_contract(settings)` calls `validate_aws_ecs_settings(settings)`; any `ContractCheck.ok is False` raises `RuntimeError` joining each failing `name: detail` plus `"Run \`elspeth doctor aws-ecs\` for full diagnostics."`; no-op otherwise.
