# AWS S3 Sink Plugin Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (- [ ]) syntax for tracking.

**Goal:** Add an `aws_s3` sink plugin that writes one Jinja-keyed S3 object (CSV/JSON/JSONL) with race-free `overwrite=False` semantics and Azure-parity audit records.

**Architecture:** A new `AWSS3Sink(BaseSink)` in `src/elspeth/plugins/sinks/aws_s3_sink.py`, auto-discovered by directory-scan (`plugins/infrastructure/discovery.py`) off `name = "aws_s3"` â€” no registry edit needed. Mirrors `AzureBlobSink` (`src/elspeth/plugins/sinks/azure_blob_sink.py`) line-for-line: Pydantic config, Jinja key templating, cumulative row buffering, per-row divert-on-serialization-failure, audit-on-every-call. Differs only where S3's API differs: no auth fields (`build_s3_client` from sibling plan 06), `put_object(..., IfNoneMatch="*")` instead of `upload_blob(overwrite=...)`.

**Tech Stack:** Python 3.13, Pydantic v2, boto3/botocore (`aws` extra, sibling plan 06), Jinja2 `SandboxedEnvironment`, pytest.

**Depends on:** `2026-07-08-aws-ecs-06-s3-source.md` for `build_s3_client(...)` in `src/elspeth/plugins/aws_s3_common.py` and the `aws` extra. Verified locally: botocore 1.43.42's `s3/service-2.json` `PutObjectRequest.IfNoneMatch` raises `412 Precondition Failed` when the key exists, or `409 ConditionalRequestConflict` on a losing concurrent race (Task 3). **Also depends on:** `2026-07-08-aws-ecs-08-s3-endpoint-gate.md` â€” `aws_s3` auto-discovers into the web composer catalog on registration, so it must not reach a web-reachable deployment before plan 08's `endpoint_url` gate lands (release-sequencing constraint, not a code change here).

**Deviations:** `AWSS3Sink`/`AWSS3SinkConfig` follow the codebase's all-caps-acronym convention (`CSVSink`, `JSONSink`, `HTTPCallRequest`, `LLMResponse`), diverging from sibling plan 06's mixed-case `AwsS3Source`/`AwsS3SourceConfig`. Kept as-is (house style wins); reconciling the pair is a cross-plan call outside this plan's scope.

**Global Constraints (verbatim from spec):**
- "Configuration must not include AWS access-key, secret-key, or session-token fields."
- "Audit records should use provider `aws_s3` and record operation, bucket, key, byte count, content hash where available, overwrite behavior, latency, and sanitized error class. They must not record credentials, presigned URLs, raw secret-bearing endpoint strings, or unbounded provider error bodies."
- "The sink should avoid client-side exists-then-put races. When overwrite is disabled, use S3 server-side conditional semantics if the selected boto3 API supports them for the intended call path. If a limitation remains, document it and test the behavior."
- `endpoint_url` field exists in config; the web-authorship rejection gate is sibling plan 08, out of scope here.
- Reviewed, not applicable: a panel finding about an unbounded `max_object_bytes` ceiling concerns sibling plan 06's *source* config; this sink has no such field.
