# Design — New 0.6.0 Examples: Concurrent Scheduling & Multi-Worker

**Date:** 2026-06-14
**Status:** Approved (brainstorming complete; ready for implementation plan)
**Author:** John Morrissey, Claude Opus 4.8
**Branch context:** `fix/plugins-subsystem-remediation` (off `release/0.6.0`)

## Purpose

ELSPETH 0.6.0 shipped two genuinely new engine behaviours that the existing
`examples/` suite does not demonstrate:

1. **Concurrent in-process token scheduling** (ADR-026 durable token scheduler;
   proven by `tests/integration/test_rc6_concurrent_pumping_proof.py`, C2). The
   scheduler keeps many token lifecycles open *simultaneously* rather than
   draining one token to terminal before starting the next.
2. **Multi-worker packs** (ADR-030, "One-Host WAL Pack"). Independent OS
   processes — one epoch-fenced leader plus N−1 claim-only followers launched
   with `elspeth join <run_id>` — cooperate on one `RUNNING` run sharing one
   WAL SQLite audit DB.

The existing `multi_source_queue` / `multi_flow` examples show *static* multi-
source fan-in but neither new behaviour. This work adds three examples that
make the new behaviours visible and (for two of them) self-evidently correct.

Non-goals: changing engine code, demonstrating multi-*host* deployment (refused
for 0.6.0 per ADR-030 D1), or Azure/OpenRouter/Docker variants. All three new
examples are keyless and hermetic (pure-data or ChaosLLM).

## The three examples

### 1. `examples/concurrent_scheduler/` — self-verifying, pure-data

**Demonstrates:** the scheduler holds multiple token lifecycles open at once.

**Shape:**
- Two named CSV sources, `stream_a` and `stream_b`, each `on_success: merged`,
  3 rows apiece (6 total).
- One aggregation barrier over `merged` (`plugin: batch_stats`,
  `trigger: {count: 6}`, `output_mode: transform`) → CSV sink.
- Entry point: plain `elspeth run --settings examples/concurrent_scheduler/settings.yaml --execute`.

**Why it proves the point:** a count-6 barrier can only fire if all six tokens
are alive concurrently. Under hypothetical one-at-a-time draining, token #1
reaches terminal before token #2 is admitted, so the barrier never accumulates
6 members — it would deadlock (or, with `batch_size=1` semantics, flush
degenerate singletons). It completes here *because* the 0.6.0 scheduler keeps
all six lifecycles open. This is the C2 rendezvous proof rendered runnable.

**Evidence at completion:** run summary shows the aggregation fired once over 6
members; the README shows a short `elspeth explain --run latest` (or a small
read-only SQL tail) confirming the single batch drew rows from *both* sources.
Exit 0 is the pass condition.

**Settings skeleton** (modelled on `batch_aggregation` + `multi_source_queue`):
```yaml
sources:
  stream_a:
    plugin: csv
    on_success: merged
    options:
      path: examples/concurrent_scheduler/input/stream_a.csv
      schema: {mode: observed}
      on_validation_failure: discard
  stream_b:
    plugin: csv
    on_success: merged
    options:
      path: examples/concurrent_scheduler/input/stream_b.csv
      schema: {mode: observed}
      on_validation_failure: discard
queues:
  merged: {}
aggregations:
- name: rendezvous
  plugin: batch_stats
  input: merged
  on_success: output
  on_error: discard
  trigger: {count: 6}
  output_mode: transform
  options:
    schema: {mode: observed}
    value_field: amount
    compute_mean: true
sinks:
  output:
    plugin: csv
    on_write_failure: discard
    options:
      path: examples/concurrent_scheduler/output/rendezvous.csv
      schema: {mode: observed}
landscape:
  url: sqlite:///examples/concurrent_scheduler/runs/audit.db
```
*(Exact `batch_stats` option set — `value_field`, `group_by`, `compute_*` — to
be finalised against the plugin schema during implementation; the count-6
rendezvous over two sources is the load-bearing part.)*

### 2. `examples/multi_worker/` — self-verifying, ChaosLLM, `run.sh` launcher

**Demonstrates:** `elspeth join` — independent processes share one running run.

**Shape:**
- `settings.yaml`: a ChaosLLM-backed `llm` transform (the `chaosllm_sentiment`
  pattern — `base_url: http://127.0.0.1:8199/v1`, `model: chaosllm/fake-gpt-4`,
  fake api_key), input ~60 rows so the run stays `RUNNING` long enough to join.
  Includes a `retry:` block and a `quarantine` sink (faults are part of the
  demo).
- `chaos_config.yaml`: modest steady-state error injection (mirror
  `chaosllm_sentiment`).
- `run.sh` choreography:
  1. Start ChaosLLM on 8199 (`--workers 1`), wait for `/health`.
  2. Clear stale `runs/*.db*` and `output/`.
  3. Launch `elspeth run --settings … --execute &` (the **leader**), background.
  4. **Poll the audit DB** (`runs` table) until a `RUNNING` run with ≥1 claimed
     `token_work_items` row exists; capture its `run_id`. Bounded retry with a
     clear timeout error if the leader never reaches RUNNING.
  5. Launch `WORKERS` (env, default `1`) followers:
     `elspeth join "$run_id" --settings … --execute &` each.
  6. `wait` for leader + all followers.
  7. **Query `token_work_items` grouped by `lease_owner`** (cross-referenced
     with `run_workers`) and print a per-worker completed-row table + wall-clock
     throughput.

**The self-verification:** the script asserts that **≥2 distinct `lease_owner`s
each completed at least one row**, and prints
`✓ PASS: leader + N follower(s) shared M rows across W workers` or a `✗ FAIL`
with diagnostics. Non-zero exit on FAIL.

**README:** leader vs follower roles, the DB-poll join-discovery mechanism (and
*why* it works — coordination state lives in the DB, ADR-030), `WORKERS` env
override, and exit-code semantics (`0` clean depart, `1` JoinRefusedError,
`3` SIGINT/eviction, `4` framework error — per slice-5 CLI contract).

**Cleanup:** `trap cleanup EXIT` kills the ChaosLLM server and any still-running
worker PIDs (mirror the `chaosllm_endurance` trap).

### 3. `examples/multi_worker_showcase/` — flashy, demonstrative-only

**Shows off:** the whole 0.6.0 engine at once. Proves nothing in isolation.

**Shape:**
- Multiple named sources fanning in + the concurrent scheduler + a pack of
  **4 workers** (1 leader + 3 followers, `WORKERS` default `3` followers) against
  ChaosLLM on a larger input (~200 rows), with retries/quarantine for visible
  drama.
- `run.sh` launches the swarm (same leader→DB-poll→join pattern as example 2,
  scaled up), tails the combined live progress of the processes, and on
  completion prints a final ASCII "stats card": workers spawned, total rows,
  aggregate rows/sec, succeeded / quarantined counts. Banner header + flourish.
- **No assertions, no PASS/FAIL.** README states explicitly: "A demo to watch,
  not a proof. For the rigorous versions see `multi_worker/` (checks worker
  sharing) and `concurrent_scheduler/` (checks concurrent rendezvous)."

## Cross-cutting conventions

Match the existing suite exactly:
- Per-example: `README.md`, `settings.yaml`, `input/` CSV(s), `output/` (git-
  ignored artifacts), `runs/` audit DB. ChaosLLM examples also get
  `chaos_config.yaml` + `run.sh` with a `trap cleanup EXIT`.
- ChaosLLM servers always started with `--workers 1` (documented errorworks bug,
  `examples/AGENTS.md`).
- Register all three in `examples/README.md` and `examples/AGENTS.md` under a new
  section: **"0.6.0 — Multi-Worker & Concurrent Scheduling."**
- Naming locked: `concurrent_scheduler`, `multi_worker`, `multi_worker_showcase`.
- Parameters locked: example 2 `WORKERS=1` default (leader + 1 follower = 2-way);
  example 3 three followers (4-way); row sizes ~60 (ex2) and ~200 (ex3).

## Components & boundaries

| Unit | Purpose | Interface | Depends on |
|------|---------|-----------|------------|
| `concurrent_scheduler/settings.yaml` + inputs | Provoke a 2-source count-6 rendezvous | `elspeth run` | csv source, `batch_stats` aggregation, csv sink |
| `multi_worker/run.sh` | Orchestrate leader + follower(s), verify sharing | bash; reads audit DB read-only for run_id + attribution | `elspeth run`, `elspeth join`, ChaosLLM, sqlite3 |
| `multi_worker/settings.yaml` + `chaos_config.yaml` | ChaosLLM-backed slow workload | shared by leader & followers (same `config_hash`) | `llm` transform, ChaosLLM |
| `multi_worker_showcase/run.sh` + config | Scaled-up spectacle | bash; tail + stats card | same as ex2, plus multi-source |
| README updates | Discoverability | docs | — |

The two `run.sh` launchers share an identical leader→poll→join→reap skeleton;
the showcase differs only in scale, the live-tail presentation, and the absence
of the assertion block. The implementation plan should factor the shared
skeleton deliberately (either a copied-and-commented block per example — the
existing suite favours self-contained scripts — or a tiny shared helper; decide
in planning, leaning self-contained to match suite convention).

## Risks & open implementation questions

1. **Join-window timing.** The follower can only attach while the run is
   `RUNNING`. At ~60 rows of ChaosLLM latency the window is comfortably wide,
   but the DB-poll loop must tolerate the leader finishing fast (degenerate
   case: leader completes before any follower joins → example 2's assertion
   would FAIL). Mitigation: size the input / chaos latency so the leader cannot
   drain 60 rows before the poll launches a follower; if the race is tight,
   raise row count rather than add artificial sleeps. Validate empirically
   during implementation.
2. **`batch_stats` exact options.** Finalise `value_field` / `group_by` /
   `compute_*` against the live plugin schema so the count-6 barrier emits a
   sensible summary row.
3. **Read-only DB access during a live run.** The attribution query must open
   the audit DB read-only (WAL reader) so it never contends with worker writes
   — consistent with slice-1's "read-only web reads" discipline. Use
   `sqlite3 file:...?mode=ro` or `PRAGMA query_only`.
4. **`run_workers` / `token_work_items` column names** for the attribution query
   to be confirmed from `src/elspeth/contracts/coordination.py` and the epoch-21
   schema during implementation (columns `lease_owner` / `scheduler_lease_owner`
   confirmed present).
5. **Dogfood/CI cost.** Example 3 (~200 rows × 4 workers) is the heaviest;
   keep it out of any fast dogfood gate (mark in AGENTS.md like
   `chaosllm_endurance`).

## Verification of the examples themselves

- `concurrent_scheduler`: `elspeth run … --execute` exits 0 and the aggregation
  fires once over 6 members from both sources.
- `multi_worker`: `./run.sh` exits 0 and prints `✓ PASS` with ≥2 workers each
  having completed ≥1 row.
- `multi_worker_showcase`: `./run.sh` exits 0 and renders the stats card; no
  correctness assertion (demonstrative).
- All three added to the README/AGENTS tables and runnable from repo root.
