## Executive summary

This is a genuinely good design: problem-first, doctrine-aligned, and its central architectural move (pull schema initialization out of app construction into a one-shot `doctor` command plus validate-only startup) correctly targets a real defect in the current code (`web/app.py:875` runs `initialize_session_schema` during app construction). The "Existing Context" section is accurate — every checkable claim verified against source. The design is not ready to emit into an implementation plan for one reason: it introduces an `endpoint_url` field on the new web-authorable `aws_s3` plugins with zero egress consideration, which — if built as specified — ships a pipeline-payload exfiltration / SSRF vector that the project already treats as a must-gate class everywhere else. Fix that control, quantify the readiness/NFR budgets, and address the Aurora-Serverless-v2 and doctor-concurrency realities, and this is emittable.

## Findings

### Critical (must-fix before emission)

**C1 — [Tech-before-problem / integration-reality] `endpoint_url` on web-authorable `aws_s3` plugins is an unguarded data-egress + SSRF vector**

- Spec evidence: lines 234–235 introduce `optional endpoint_url for LocalStack or tests` with no security discussion; lines 237–240 correctly forbid access-key/secret-key fields and mandate the AWS default credential chain (in ECS: the **task role**). The combination — author-controlled endpoint + server-ambient credentials — is exactly the asymmetry the project gates elsewhere.
- Code evidence that this is live doctrine, not theory:
  - `src/elspeth/web/provider_config_policy.py:111` (`web_llm_base_url_policy_error`) rejects a web-authored `base_url` on LLM nodes precisely because "the api_key is resolved server-side … a custom base_url … would direct the server-held bearer credential to an author-chosen destination (a credential-egress / SSRF path)."
  - `provider_config_policy.py:18` (`MANAGED_IDENTITY_POLICY_ERROR`) blocks web-authored RAG configs from using a server credential against an author-influenced target.
  - These gates are wired **only** for transforms: `web/execution/validation.py:1480,1525,1580` and `web/composer/tools/transforms.py`. There is **no equivalent gate for source/sink endpoint fields.**
- Composer reachability (why this is Critical, not theoretical): the composer hides plugins only when they declare *required secret fields* — `web/composer/tools/_availability.py:12` `filter_secret_available_summaries`. `aws_s3`, by design, declares **no** secret fields, so it is **not** hidden; `list_sinks`/`list_sources` surface it (`transforms.py:129`, `sources.py:91`) and `set_output` only existence-checks the plugin name (`web/composer/tools/outputs.py:136` `_validate_plugin_name`). Any authenticated composer author can therefore configure an `aws_s3` sink with an arbitrary `endpoint_url`.
- Impact (framed on the airtight axes — note: SigV4 does **not** transmit the secret key, so this is not credential theft; the signature is Host-scoped and non-replayable):
  - **Sink = data exfiltration (crown jewel).** The sink serializes pipeline output and PUTs it as the request body to the author-chosen host. For a government data-pipeline tool whose whole posture is controlled egress + redaction, a web-authorable config that ships pipeline payloads to an arbitrary host is Critical on its own. (The `x-amz-security-token` and access-key-id also travel there; they cannot be replayed against real S3, but their disclosure is an unnecessary bonus leak.)
  - **Source = SSRF / response injection.** The server issues signed requests to an author-chosen host and ingests the reply as rows — internal host/port probing plus attacker-controlled data injected into the pipeline.
- Framing: this is a **design-level** gap — `aws_s3` does not exist yet (verified: `src/elspeth/plugins/sources/` and `sinks/` contain `azure_blob_*` but no `aws_s3_*`). The finding is "the design omits a control the codebase's own doctrine requires," which is the honest and stronger framing.
- Recommendation: Add an explicit egress control to the design, at the seam that already hosts the sibling gate (`web/execution/validation.py`, alongside `web_llm_base_url_policy_error`). Two acceptable shapes, mirroring existing precedent: (a) **forbid** a web-authored `endpoint_url` outright — canonical AWS endpoints only; LocalStack/private endpoints become an operator-runtime concern (mirrors the LLM `base_url` gate and its "private gateway = operator runtime, not web-author option" rationale); or (b) a **suffix-floor allowlist** validator as used in the prior `vault_url` SSRF fix. Add a corresponding Testing-Strategy line ("web-authored `aws_s3` with a non-canonical `endpoint_url` is rejected") and an Acceptance criterion. Keep `endpoint_url` available for CLI/batch authorship where the single-machine threat model holds — exactly the split `provider_config_policy` already documents.

### High

**H1 — [NFR handwaving + integration-reality] Zero quantified NFRs; unauthenticated `/api/ready` with live probes is a DoS/latency amplifier; object reads are unbounded**

- Spec evidence: no startup-time budget, no readiness/DB-probe timeout budget, no S3 object-size cap, no Aurora connection-pool sizing anywhere in the document. `/api/ready` (lines 162–182) performs session-DB + landscape-DB connectivity, two schema validations, and three filesystem writability probes per call.
- Code evidence: `/api/health` is unauthenticated today (`web/app.py:1157`, no `Depends(get_current_user)`); a parallel unauthenticated `/api/ready` that does live dependency probes has no rate-limit, cache, or timeout specified. The azure_blob source reads whole objects into memory unbounded (`azure_blob_source.py:486` `download_blob().readall()`); `aws_s3` "equivalent in capability" (spec line 17) inherits this — `get_object()["Body"].read()` with no size ceiling on an author-named object.
- Amplifiers behind an ALB:
  - **DB load:** each probe opens real connections to both Aurora targets.
  - **EFS write churn:** each `/api/ready` hit creates+fsyncs+deletes three probe files on EFS (data_dir, payload store, blob dir) — constant network-fsync latency and IOPS cost, not just DB load, at ALB poll cadence.
- Impact: an unauthenticated readiness endpoint that touches every dependency on every call, with no budget, is both a self-inflicted latency tax and an external DoS surface; unbounded object reads make a single large author-named S3 object an OOM lever on the single web task.
- Recommendation: Add quantified NFRs — readiness per-check timeout budget (e.g. per-probe ≤ N ms, whole-endpoint ceiling), a short readiness result cache (e.g. 1–5 s) so ALB polling can't stampede dependencies, an `aws_s3` max-object-size guard with a fail-closed error, and Aurora pool sizing. Decide and state whether `/api/ready` is authenticated or rate-limited. State the startup-time budget the validate-only path must meet.

**H2 — [Integration-reality gap] "Aurora Serverless v2" is named but treated as generic PostgreSQL**

- Spec evidence: lines 11, 27, 96–100, 344 assume only a PostgreSQL SQLAlchemy scheme; nothing addresses Serverless-v2-specific behavior. The design mandates live DB connectivity at startup (validate-only) and in readiness, with no connection-timeout, retry, or backoff contract.
- Impact (framed conditionally — Aurora auto-pause/scale-to-zero is **opt-in**, min-capacity = 0 ACU): *if* scale-to-zero is configured, the first connection after an idle pause incurs a multi-second cold start. A readiness probe with no timeout budget will flap during that window (ALB pulls the task), and the validate-only startup connection can time out on a cold cluster. Conversely, liveness/readiness probes that keep opening connections can hold the cluster awake indefinitely, defeating auto-pause and its cost rationale. Even without auto-pause, the absence of any connection-timeout/backoff contract is a production-readiness gap.
- Recommendation: Add an explicit Aurora section: connection timeout + bounded retry/backoff for the startup validate path, a readiness-probe timeout that tolerates a single cold-start without flapping (or an explicit "auto-pause unsupported for this milestone" non-goal), and a statement on pool warm-up. Keep the concern conditional on the auto-pause config so it can't be waved off.

### Medium

**M1 — [Migration/transition thinking] `doctor --init-schema` has no concurrency contract for the genuinely racy window**

- Spec evidence: lines 144–155 describe `--init-schema` as "may create missing schema objects"; the transition model (one-shot doctor task before the web task takes traffic) is coherent for the simple ordering. Credit where due: the validate-only startup with "schema missing → fail → run doctor" (lines 153–155, Acceptance line 351) already handles doctor-then-web ordering gracefully and fail-closed.
- Gap: the spec is silent on the two windows that actually race — two concurrent `--init-schema` invocations (deploy retry, parallel deploys) and a doctor mid-init while the web task validates. Postgres `metadata.create_all()` run concurrently by two creators can raise duplicate/"tuple concurrently updated" errors or deadlock; there is no advisory-lock or serialization story.
- Recommendation: Specify a Postgres advisory lock (e.g. `pg_advisory_lock`) around the init path, or state the operational invariant that exactly one init task runs at a time and how ECS guarantees it. Add a Testing-Strategy line for concurrent `--init-schema`.

**M2 — [Integration-reality] SQLite-on-EFS locking/corruption hazard for local auth is unstated**

- Spec evidence: lines 193–194, 202–204, 357 place the local-auth SQLite DB on EFS as an explicit single-task option. Code confirms the path: `web/app.py:832` `LocalAuthProvider(db_path=settings.data_dir / "auth.db", ...)` — under the EFS mount in aws-ecs.
- Gap: SQLite over NFS-class filesystems is explicitly warned against by both SQLite's own docs and AWS for concurrent access (byte-range lock semantics differ; corruption risk). The single-task posture mitigates steady state, but the doctor one-shot task and the web task can both be alive during a deploy, and each can open `auth.db`. The spec neither names the hazard, mandates a locking mode, nor scopes the doctor's `auth.db` access. Partially mitigated by the design's own "not a multi-task identity design" caveat and Cognito/OIDC recommendation (lines 195, 202–204, 358) — good hedges, but the corruption class is not called out.
- Recommendation: Name the hazard; state that `doctor` must not open `auth.db` (or must do so under the same locking discipline), and record the SQLite-journal/locking mode assumption. Reinforce that OIDC/Cognito is not merely "recommended" but the only concurrency-safe option once more than one task can exist.

**M3 — [Design coherence] Four overlapping probe surfaces with different DB contracts, and no ECS wiring contract**

- Spec evidence + code: the design now yields four probes with *different* semantics — CLI `elspeth health` (checks env `DATABASE_URL`, which is unset in aws-ecs → silently "skip", `cli.py:3044-3067`), CLI `doctor aws-ecs` (web session/landscape URLs), HTTP `/api/health` (shallow `{"status":"ok"}`, `app.py:1157`), and HTTP `/api/ready` (full deps). The spec never says which ECS mechanism calls which.
- Impact: the spec's own liveness/readiness split (its stated reason for `/api/ready`, lines 180–182) is defeated if the wiring is left implicit. Wire the ECS *container* healthCheck to `/api/ready` and you reintroduce the dependency-induced restart storms the spec explicitly wants to avoid; wire it to `elspeth health` and you get a green probe that validates nothing in aws-ecs mode.
- Recommendation: State the wiring contract in the design: container healthCheck → `/api/health`; ALB target-group healthCheck → `/api/ready`; `doctor aws-ecs` → one-shot pre-traffic task; and note that `elspeth health` is not the ECS web probe (its `DATABASE_URL` contract is orthogonal to the web URLs — the spec already flags this at lines 66–68; carry it into the wiring guidance).

**M4 — [Falsifiability] The Bedrock section is a deferred investigation, and its acceptance criterion is not falsifiable**

- Spec evidence: lines 254–284 are framed as "verification-first" ("confirm…", "verify whether…"). Acceptance line 362 "Bedrock support is verified through LiteLLM for the web composer" states no observable pass/fail.
- Reality check: the composer already routes through LiteLLM (`web/composer/service.py` `_litellm_acompletion`, `boot_probe.py:48`), so the composer path is plausibly config-only. But the pipeline path answer is essentially predetermined: `_PROVIDERS` (`plugins/transforms/llm/transform.py:251`) contains only `azure` (Azure SDK) and `openrouter` (HTTP) — there is **no** existing generic LiteLLM pipeline provider — so "verify whether the current registry can route Bedrock through an existing LiteLLM path" resolves to "no; a new provider is required." The "verification-first" framing under-commits relative to a known answer.
- Recommendation: Either commit the design to "add a LiteLLM-backed `bedrock` pipeline provider registered in `_PROVIDERS`" (note the discriminated-union constraint `len(_PROVIDERS) >= 2` at `transform.py:1179` — adding a third variant is fine), or explicitly defer Bedrock to a follow-up. Make the composer acceptance falsifiable: e.g. "unit tests assert Bedrock-shaped LiteLLM responses parse token/cost/prompt-cache metadata; a live-gated smoke test completes one `bedrock/anthropic...` request."

**M5 — [NFR handwaving] "must not be known placeholders" — the placeholder inventory is undefined**

- Spec evidence: line 105 requires production secrets "must not be known placeholders"; Testing line 311 and Acceptance imply "production secret placeholder rejection" but never define the placeholder set or which secret-backed settings are covered.
- Reality check: the codebase implements exactly two ad-hoc heuristics, both gated on host (`_allow_insecure_test_keys`, active in aws-ecs since host = 0.0.0.0): `secret_key == "change-me-in-production"` and length floor (`config.py:603-618`), and uniform-byte detection for `shareable_link_signing_key` (`config.py:620-644`). No placeholder detection exists for `landscape_passphrase`, OIDC secrets, or provider API keys.
- Recommendation: Enumerate in the design which secret-backed settings the aws-ecs contract validates and against what placeholder inventory (reference the two existing heuristics and state whether the set is extended). Otherwise "placeholder rejection" is an untestable acceptance criterion.

**M6 — [Traceability] Two Scope bullets are under-covered in Testing/Acceptance**

- `/api/health` liveness independence: Acceptance line 354 asserts `/api/health` "remains shallow liveness," but no Testing line asserts that liveness does **not** touch Aurora/EFS/providers (lines 159–161 state the requirement; nothing verifies it). Add a test: `/api/health` returns 200 with dependencies down.
- `endpoint_url` handling: the Scope bullet "aws_s3 … plugins" (line 30) lands in Testing (config/format/overwrite/audit, lines 317–321) and Acceptance (lines 359–361), but `endpoint_url` — the C1 field — has no test or acceptance line at all. Add both (ties to C1's recommended gate).
- Otherwise, Scope→Testing→Acceptance mapping is strong and mostly complete; this is a targeted gap, not a systemic traceability failure.

## What the design does well (genuine strengths)

- **Problem-first framing and honest non-goals.** Purpose and Scope precede any tech commitment; Non-Goals (lines 35–52) explicitly carve out Terraform, SQLite→Aurora migration, multi-container scaling, and S3-backed internal blobs, each with a reason. Follow-Up Work (367–378) separates the genuinely separable designs rather than smuggling them in.
- **Correctly targets a real code defect.** The core move — schema init out of app construction (verified at `web/app.py:875`) into one-shot `doctor` + validate-only startup — is the right fix, and the fail-closed "schema missing → fail → run doctor" transition is coherent.
- **Doctrine-aligned throughout.** Drop/recreate as the pre-1.0 migration model, single-web-process as an acknowledged deferred limit (verified: `web/app.py:944-975` hard-refuses multi-worker), and one-instance-per-org isolation are all respected — no findings manufactured against deliberate decisions.
- **Pragmatic EFS validation.** Lines 197–200 correctly refuse to try to *detect* EFS and instead prove the runtime contract Elspeth needs (create/read/fsync/delete a probe file as the non-root user, without leaking contents) — the right instinct.
- **Liveness/readiness split with redaction carried through.** Structured per-check readiness with "no secrets or credential-bearing URLs" (lines 168–169, 355) and the Security-and-boundary section (335–340) invoking Wardline for new boundaries and demanding redaction in doctor/readiness/plugin surfaces. (Supporting context, not a finding: the existing CLI `elspeth health` echoes raw `str(e)` on DB errors, `cli.py:3057-3061` — a redaction footgun `/api/ready` must not repeat; the spec's redaction requirement already covers this.)
- **`aws_s3` credential hygiene is right.** Omitting access-key/secret-key fields, using the default credential chain / task role, and calling for S3 server-side conditional writes to avoid client-side exists-then-put TOCTOU races (lines 247–250) is exactly the correct pattern.

## Confidence Assessment

- **High confidence** in the reality-check verdicts (session/landscape URL split, SQLite defaults, in-app-construction schema init, `elspeth health`/`DATABASE_URL` contract, multi-worker refusal, azure_blob source/sink parity, composer-on-LiteLLM, provider-specific `_PROVIDERS`). Each is cited to specific file:line and read directly.
- **High confidence** in C1's exploit chain to the composer boundary (no secret fields → not hidden by `filter_secret_available_summaries` → `set_output` existence-check only), and in the corrected SigV4 mechanism (data-exfiltration/SSRF, not credential theft).
- **Medium confidence** on the precise blast radius of H2 (Aurora auto-pause behavior) and M2 (EFS+SQLite corruption likelihood) — these depend on operator-chosen configuration (min-ACU, EFS mount options, deploy overlap) not fixed by the spec; findings are framed conditionally for that reason.

## Risk Assessment

- **Highest residual risk if emitted unchanged:** C1 — an implementation plan built from this spec would add a web-authorable payload-exfiltration/SSRF surface, directly contradicting the product's controlled-egress posture. This is the emission blocker.
- **Operational risk:** H1/H2/M3 together mean the deployment could pass functional tests yet flap, self-DoS, or run a probe that validates nothing in production; these are the "green in CI, red in prod" class.
- **Low risk of over-correction:** the recommended C1 fix reuses an existing, proven seam (`provider_config_policy` / `web/execution/validation.py`) and pattern (LLM `base_url` gate, `vault_url` suffix-floor), so remediation cost is low and well-precedented.

## Information Gaps

- `aws_s3` source/sink, the `doctor aws-ecs` CLI group, and `/api/ready` **do not exist yet** (verified absent). Findings C1, H1 (readiness/object-size), M1, M3(`/api/ready`), M4(pipeline provider) are therefore **design-level assessments** against existing code patterns plus documented boto3/SigV4/Aurora/EFS behavior — not verified runtime facts.
- The ECS task-definition / ALB target-group configuration (which probe is wired where, Aurora min-ACU, EFS access-point uid/gid mapping) lives in Terraform, which is explicitly out of repo and out of scope — several findings (H2, M2, M3) can only be fully closed against that artifact, which was not available for review.
- I did not exhaustively trace whether any non-composer surface further restricts `aws_s3` authorability; I verified the two gates that matter (secret-availability filter, `set_output` plugin validation) and found no allowlist, but did not read every catalog code path.

## Caveats

- This is a targeted single-artifact review of a brainstorm-to-spec design document, not a numbered solution-architecture package. Per the review scope, I did not penalize the absence of RTM/ArchiMate/ADR-set artifacts the format never promised; I assessed only the canonical failure modes that apply to any design.
- Severity of C1 is stated on the project's own threat model, in which authenticated composer authors are untrusted with respect to server-held credentials (the premise of the entire `provider_config_policy` module). If, contrary to my verification, `aws_s3` is intended to be CLI/batch-only and never composer-authorable, C1 drops to High (defense-in-depth) — but the code shows no such restriction today, so it stands as Critical.
- The `str(e)` DB-error behavior in the existing CLI `elspeth health` is noted as supporting context for a redaction footgun the new `/api/ready` must avoid; it is not scored as a finding against this spec.
