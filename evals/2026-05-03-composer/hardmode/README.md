## Persona specifications (`personas/`)

Each spec is what the persona-subagent was given at spawn time. Sections common to all three:
- **Bio** — a 2-3 sentence backstory grounding the persona's domain experience and tooling familiarity
- **Cognitive style** — how this persona thinks, prioritises, and writes
- **Linguistic constraints** — must-use moves and must-avoid vocabulary (these are what kept the subagent in character)
- **Knowledge gaps and misconceptions** — what the persona believes about the product that may or may not be true
- **Stop conditions** — explicit DONE-trigger criteria

|---|---|---|
| `p1_compliance.md` | Linda Marston, Senior Compliance Officer | Over-constrained — hedges and conditions before the ask |

## Scenario fixtures (`scenarios/`)

Each `*.json` is the immutable definition of one scenario. Fields:
- `scenario_id` — short ID matching the `results/` subdirectory name
- `persona` — which persona spec drives the dialogue
- `task_class` — `happy` | `edge` | `limit`
- `task_summary` — one-line description
- `csv_filename`, `csv_content` — embedded CSV for scenarios that upload data; `null` for the limit scenarios that test asks-without-data
- `opening_prompt` — the literal text the persona's first message contains (turn 1 user input is hardcoded; turn 2+ is generated by the persona-subagent in real time)
- `probe` — a comment field describing what failure mode this scenario is designed to surface
- `product_capability_used` — what features the scenario exercises (or "NONE" for refusal probes)
- `expected_outcome` — the operator's prediction
- `pass_criterion` — what counts as a pass

**3 × 3 design** — each persona × each task class:

| | Happy (aligned with strengths) | Edge (combines patterns) | Limit (refusal probe) |
|---|---|---|---|
| **P1 Linda** | `p1_t1_happy_categorize` | `p1_t2_edge_multiregion` | `p1_t3_limit_sharepoint` |
| **P2 Sarah** | `p2_t1_happy_themes` | `p2_t2_edge_cooccurrence` | `p2_t3_limit_pdf` |
| **P3 Marcus** | `p3_t1_happy_lead_score` | `p3_t2_edge_enrichment` | `p3_t3_limit_streaming` |

## Per-scenario results (`results/<scenario_id>/`)

Each scenario subdirectory holds a complete run-record. The exact file inventory varies by outcome class, but the canonical full set is:

### Bootstrap (always present)
| File | Contents |
|---|---|
| `scenario.json` | Copy of the fixture (immutable record of what was tested) |
| `session.json` | Composer's response to `POST /api/sessions` (session ID, title) |
| `sid.txt` | Bare session ID for shell convenience |
| `jwt.txt` | The JWT used for that scenario's HTTP calls |

### CSV-upload scenarios (7 of 9 — limit-probes P1-T3 and P3-T3 had no CSV by design)
| File | Contents |
|---|---|
| `blob.req.json` | Request body sent to `/api/sessions/{sid}/blobs/inline` (filename + mime_type + content) |
| `blob.json` | Response — blob ID, size_bytes, created_by, storage_path |

### Per turn N (always at least N=1, may go up to N=3)
| File | Contents |
|---|---|
| `turn{N}.user.txt` | Literal text the persona sent (turn 1: from fixture; turn 2+: from the persona-subagent) |
| `msg.t{N}.req.json` | The JSON envelope that wrapped the user text into the `/messages` POST |
| `msg.t{N}.resp.json` | Composer's full response (`.message.content` for assistant text, `.state` for resulting composition state) |
| `msg.t{N}.curl_meta` | HTTP code + wall-clock seconds (`200 75.48` style) |
| `state.before.t{N}.json` | Composition state before this turn (for mutation diff) |
| `state.after.t{N}.json` | Composition state after this turn (version, is_valid, validation_errors, nodes) |
| `progress.t{N}.json` | Snapshot of `/composer-progress` events post-call (often empty post-completion) |
| `metrics.t{N}.json` | Per-turn process metrics: tool_call_count, in_loop_recovery_count, mutated_state, is_valid_after, asked_clarifying_question, volunteered_limit, wall_seconds, assistant_content_excerpt |

### After persona DONE (only for scenarios that converged to a state)
| File | Contents |
|---|---|
| `validate.json` | Composer's `/validate` response — the 9-check matrix |
| `final_yaml.json` | The actual YAML the composer rendered (the load-bearing config) |
| `messages.json` | Full conversation history as the API records it |

### After execute (only for scenarios that reached `/execute`)
| File | Contents |
|---|---|
| `execute.json`, `execute.code` | `/execute` response + HTTP status code |
| `run.json` | Final `/api/runs/{rid}` row (the audit-record summary) |
| `diagnostics.json` | Per-token, per-state diagnostics from the engine |

### Always
| File | Contents |
|---|---|
| `ledger.json` | Per-scenario summary written by `finalize_scenario.sh`: outcome, turn count, total_tool_calls, total_in_loop_recoveries, mutating_turns, clarifying_question_turns, volunteered_limit_turns, run_status, headlines |

## Harness scripts

Three bash scripts that drive the harness. Each is independently runnable:

### `harness.sh <scenario_id>`
Bootstrap. Reads `scenarios/<scenario_id>_*.json`, logs in fresh, creates a composer session, uploads the CSV blob if the scenario has one. Writes `session.json`, `sid.txt`, `jwt.txt`, `blob.json` (if applicable) into `results/<scenario_id>/`. Does not send any messages.

### `post_message.sh <scenario_id> <turn_index> <user_message_file>`
Per-turn driver. Captures state-before, posts the user message to `/messages`, captures composer response, captures state-after, captures `/composer-progress` events, computes per-turn metrics. Writes `msg.t{N}.{req,resp,curl_meta}`, `state.{before,after}.t{N}.json`, `progress.t{N}.json`, `metrics.t{N}.json`.

### `finalize_scenario.sh <scenario_id>`
After-DONE finaliser. Calls `/validate`. If valid, calls `/execute` and waits for the run to land. Writes `validate.json`, `execute.{json,code}`, `run.json`, `diagnostics.json`, `final_yaml.json`, `messages.json`. Then aggregates everything (including all `metrics.t*.json`) into `ledger.json`.

The persona-subagent decisions (turn 2+ user messages) are not in these scripts — they're produced by the parent agent (Claude in the main thread) spawning a `general-purpose` subagent per turn with the persona spec + conversation history. The parent agent writes the subagent's reply into `turn{N}.user.txt` and then calls `post_message.sh` for that turn.

## Aggregate summary (`aggregate.json`)

Cross-scenario scorecard generated after all 9 scenarios completed. One entry per scenario with: id, persona, task_class, user_turns, engine_ran, run_status, convergence_timeout flag, and a one-line headline. Useful for at-a-glance answers to questions like "which scenarios reached the engine?" or "which timed out?"

## How to verify any claim in the companion documents

If a claim in the boss-facing brief, the technical eval report, or the basic-mode cross-reference is unclear or seems too strong, the verification path is:

1. **Find the scenario** from the claim's context (P1-T1, P2-T2, etc.).
2. **Read `scenario.json`** in that subdirectory to confirm what was tested.
3. **Read `turn{N}.user.txt` files** to see the literal persona-side messages in order.
4. **Read `msg.t{N}.resp.json`** with `jq -r '.message.content'` to see the composer's literal responses.
5. **Read `final_yaml.json`** to see the actual YAML config the composer produced.
6. **Read `validate.json` and `run.json`** for the engine outcome.
7. **Read `diagnostics.json`** for per-row engine errors with the verbatim exception text.
8. **Read `ledger.json`** for the consolidated summary.

Two specific verification queries worth knowing:

```bash
# Confirm the model the composer chose for any happy-path scenario
jq -r '.yaml' results/p1_t1_happy/final_yaml.json | grep -E '^\s+model:'

# Confirm the engine error from any failed run
jq '.. | objects | select(.error?) | .error' results/p1_t1_happy/diagnostics.json

# Confirm the proof-of-fix output
cat /home/john/elspeth/data/outputs/q3_regional_compliance_categories.csv
```
