# Code Review: Orchestrator Decomposition (6-file `engine/orchestrator/` package)

## 1. Executive Summary

The decomposition is healthy. The extraction is faithful (multiple findings confirm the new modules are byte-for-byte or behavior-preserving moves of monolith code), and crucially it made several pre-existing latent defects *reviewable per-unit* that the monolith hid — that is the decomposition working as intended, not regressions it introduced. Of the confirmed findings: **2 critical/high real bugs** (1 critical RUNNING-before-gates resume defect, plus a high `tokens[-1]` IndexError), **3 additional high-severity issues** (2 silent-failures, 1 type-design), **7 medium**, and the remainder **low** (smells, quick-wins, type-design nits). No finding indicts the decomposition's structure itself; the recurring theme is *audit-record fidelity on error/interrupt paths*. Every "real bug" below is on the audit legal-record path, so fixes must be applied as reviewed correctness changes, not drive-by cleanups. **Note on every suggested fix: where the original finding's `suggested_fix` was mechanically broken or behavior-unsafe, this report carries the verdict-corrected fix instead.**

## 2. Real Bugs (bug / silent-failure)

Ordered by severity. Includes critical/high plus the medium and low bug/silent-failure findings (task: drop nothing confirmed).

### CRITICAL

**[resume.py:420 / 547 / 568] RUNNING status committed before integrity gates; failed gate strands the run permanently** *(merge of two findings: critical source + high source — same bug)*
`reconstruct_resume_state()` is called at resume.py:547, **outside** the `try` that opens at line 568. Inside it, `update_run_status(run_id, RunStatus.RUNNING)` is a real DB commit at line 420, *before* three explicitly-checked, operator-facing gates that can raise: `checkpoint_manager is None` (433), schema contract missing (459), runtime VAL manifest drift (476) — plus uncaught DB I/O from `get_unprocessed_row_data` (484) / `get_incomplete_tokens_by_row` (489). Any raise in this window bypasses the broad `except Exception` (787) whose own comment says it exists "to prevent the run from being stuck in RUNNING permanently." Verified via caller chain: no outer finalizer (CLI `_orchestrator_context` has only a resource-closing `finally`, no FAILED-finalization), and `recovery.py:192-193` refuses any future resume of a RUNNING run. The run is genuinely terminally wedged with no terminal ceremony — an audit-completeness violation plus recovery-blocking state corruption.
Compounding ordering defect: `handle_incomplete_batches` (417) mutates the DB (EXECUTING→FAILED, retries) *before* the feasibility gates at 432/459/476.
- **Fix (corrected — do NOT use the finding's primary suggestion):** Both verdicts independently debunk "move `reconstruct_resume_state` inside the try" — `factory`/`run_id`/`resume_start_time` are bound *after* line 547, so the except handler hits `UnboundLocalError`, which `best_effort` swallows, leaving the run still RUNNING. The working fix: **defer the `update_run_status(RUNNING)` write until after the checkpoint-manager/contract/manifest gates all pass**, and move those gates *ahead* of `handle_incomplete_batches` to fix the irreversible-mutation-before-feasibility-gate half. Add a test asserting guard-failure paths leave run status non-RUNNING.
- **Behavior-safe:** No. This changes *when* RUNNING is recorded — that is the intended correction (current behavior is the bug), but it must ship as a reviewed correctness-fix with test coverage, not a quick win.

### HIGH

**[checkpointing.py:182] IndexError on counter-only aggregation node: `tokens[-1]` on empty tuple**
After the guard `if aggregation_state.nodes:` (180), line 182 does `agg_node_state.tokens[-1].token_id` unconditionally on the first node. `AggregationNodeCheckpoint.tokens` is explicitly allowed to be empty (`()`) — confirmed by the producer (`executors/aggregation.py` emits a node when `has_tokens OR has_counters`, with `token_checkpoints=()` for the counter-only case) and by `__post_init__`'s dedicated empty-tokens branch. A run interrupted immediately after a flush (the natural checkpoint-then-shutdown sequence, counter-only node first in dict order) raises `IndexError`. This propagates non-gracefully (`checkpoint_interrupted_progress` is called at `run_core.py:552` *before* the `GracefulShutdownError` raise at 558; `core.py` catches only `GracefulShutdownError`), so the run crashes AND the resume marker is never written — resumability lost.
- **Fix (corrected):** Do NOT use the finding's Variant A (guard `tokens` and leave `checkpoint_agg_state=None` → drops the durable counters the snapshot exists to preserve). Instead: keep `checkpoint_agg_state = aggregation_state` whenever `aggregation_state.nodes` is non-empty (counters always persist), and guard only the `token_id`/`node_id` *anchor* resolution — when the selected node has empty tokens, skip `tokens[-1]` and let the coalesce / pending-sink / `last_token_id`+`source_id` fallback (lines 185-202) supply the anchor (ideally scan for a node *with* tokens before falling back).
- **Behavior-safe:** No — changes what is recorded on the resume/audit path; ship as reviewed correctness-fix.

**[cleanup.py:120-124] Cleanup `RuntimeError` masks the in-flight run failure, dropping its partial-result ceremony and counters**
`cleanup_plugins` runs in the `finally` of both `_execute_run` (core.py:1004) and `process_resumed_rows` (resume.py:909). When a teardown hook fails *and* a run exception is already propagating, line 123 does `raise RuntimeError(...) from pending_exc` — in a `finally` this *replaces* the in-flight exception. A `_RunFailedWithPartialResultError` (carrying real partial counters) is supplanted, so `run()` no longer matches `except _RunFailedWithPartialResultError` (core.py:645) and lands in the generic `except Exception` (679), where `emit_failed_ceremony(result=None)` fabricates an all-zero RunResult. The FAILED RunSummary records zeroed terminal metrics and "Plugin cleanup failed" as the cause instead of the real pipeline error — silent corruption of the legal record. (Tier-1 errors bypass this; ordinary cleanup-hook failures trigger it. No existing test pins the cleanup-error-AND-pending-exception interaction.)
- **Fix (corrected):** `raise pending_exc` (the finding's first option) trades one swallow for another — the cleanup failure then survives only as a logger.warning. Prefer attaching `cleanup_errors` as a *non-replacing annotation* on `_RunFailedWithPartialResultError` so the 645/778 dispatch still fires AND the cleanup error stays audit-visible. This requires the error type to carry cleanup metadata and the ceremony to record it — more than a one-liner.
- **Behavior-safe:** No — changes the legal record (zeroed → real partial counters). Audit-owner gated.

### MEDIUM

**[ceremony.py:88-109] `safe_flush_telemetry` only catches `TelemetryExporterError`; `flush()` can raise arbitrary exceptions that mask the original run exception**
`TelemetryManager.flush()` re-raises whatever is in `_stored_exception` (manager.py:520-526), which the broad `except Exception` background-thread handler (188-191) can set to a *non*-`TelemetryExporterError`. Since `safe_flush_telemetry` is called from `finally` blocks (core.py:712, resume.py:795), a programming-error escaping it masks the in-flight run exception — exactly the masking the function exists to prevent. The codebase already has the canonical fix shape: `engine/_best_effort.py` (used 18+ times), and ceremony.py already imports it.
- **Fix:** Broaden the catch to `except Exception`, preserving the `if pending_exc is None: raise` gate. Prefer routing the broad arm through the existing `best_effort` helper (inherits its allowlist entry / logging-policy compliance) and log `type(e).__name__` per the existing idiom rather than `sys.exc_info()[1]`.
- **Behavior-safe:** Yes — changes behavior *only* in the masking scenario, which is the bug; preserves the run exception (audit-primary over telemetry).

**[ceremony.py:124-135 / 153-211] `best_effort` wraps `finalize_run` in `emit_interrupted_ceremony` AND `emit_failed_ceremony` — audit-write failure silently swallowed** *(two findings, same structural defect)*
Both ceremonies call `factory.run_lifecycle.finalize_run(...)` (the audit write recording the terminal INTERRUPTED/FAILED status) as their first statement, and all call sites (core.py:642, 672, 709; resume.py:775, 780, 792) wrap the *entire* call in `best_effort`, which swallows all exceptions to a single logger.warning. An `AuditIntegrityError` (`@tier_1_error`, reachable on DB corruption) during `finalize_run` is silently swallowed: the terminal status is never written, the run is left RUNNING/stranded, and resume.py:788-789's own comment ("to prevent the run from being stuck in RUNNING permanently") is silently defeated. Asymmetry confirms it is unintentional: the success-path `finalize_run` (core.py:587) is *un*-guarded and surfaces loudly.
- **Fix (corrected):** Do NOT apply the mechanical "move `finalize_run` outside best_effort" — it lets the finalize exception *replace* the in-flight `GracefulShutdownError` / `failed_exc.original_error` (load-bearing on the audit path), changes CLI exit codes, and the suggested shared `_finalize_and_emit` helper conflates the PARTIAL (exit 1) vs FAILED (exit 2) branches. A correct fix surfaces the finalize-write failure while still re-raising the *original* in-flight error (chained), applied coherently across both ceremonies and all call sites.
- **Behavior-safe:** No — alters what is recorded vs propagated. File as a tracked bug; operator-reviewed.

**[checkpointing.py:204-216] Shutdown checkpoint silently skipped (warning only) when no token/node id resolves**
`checkpoint_interrupted_progress` is the explicit-shutdown recovery path that deliberately checkpoints even when frequency would skip. But a *genuinely-resumable* coalesce state is silently dropped: at 185-190 the coalesce branch sets `node_id` unconditionally but sets `token_id` only if `pending_entry.branches` is non-empty. A pending entry with empty `branches` but non-empty `lost_branches` (created by `coalesce_executor.py:1513-1518`, retained under quorum/best_effort) is resumable yet yields `token_id=None`; if `last_token_id` is also None, line 204 fires the warning and returns *without writing a checkpoint*. Only a logger.warning records the loss, not the audit trail.
*(Corrections to the finding: the all-empty case it enumerated is benign restart-from-zero; the audit-contradiction framing is overstated — no field asserts "resumable"; and the line-182 IndexError it surfaced is filed separately as HIGH above.)*
- **Fix:** Resolve a valid `token_id` for the empty-branches pending entry (e.g., fall back to the coalesce `node_id` with a row-derived token so the barrier persists). Decide with the audit owner whether the no-resolvable case is benign or an invariant violation.
- **Behavior-safe:** No — changes what is checkpointed; audit-owner gated.

**[resume.py:680-720] `rows_coalesce_failed` from run-1 (pre-interrupt) unrecoverable on resume — count undercounts in the legal record** *(merge: medium silent-failure + low smell; kept at MEDIUM)*
`derive_resume_terminal_status_from_audit` has no match arm for `rows_coalesce_failed` because the coalesce-failure roll-up is emitted to telemetry only (`outcomes.py:462/510`), never as a queryable `token_outcomes` signal. The resume path grafts only *this-resume* failures (resume.py:720); the no-rows branch ships 0. So a run with a run-1 coalesce-quorum failure that is interrupted then resumed-to-completion reports a terminal RunResult whose `rows_coalesce_failed` undercounts vs an uninterrupted run. Bounded: terminal *status* stays correct (co-incremented `rows_failed` forces COMPLETED_WITH_FAILURES); only the roll-up count is wrong. The ~80-line inline comment (resume.py:640-719) already documents the root cause.
- **Fix:** Operator follow-up — make `rows_coalesce_failed` reconstructable from a queryable audit signal (schema/epoch change). **Do not patch the count locally** (would break the during-resume audit-counter parity the graft restores).
- **Behavior-safe:** The eventual schema fix is `changes-behavior` (alters what is persisted); the *recommended near-term action* (file it, don't touch the graft) is inert.

## 3. Architectural Smells

- **[core.py:837-868 / resume.py:127-157] (MEDIUM) Run-path and resume-path duplicate the entire four-call validation cluster verbatim**, including the `SimpleNamespace(on_write_failure=...)` stub construction. Divergent validation between run and resume is an audit correctness hazard. **Fix:** extract `validate_pipeline_config(config)` into `validation.py` performing the four `validate_*` calls + stub construction, call from both sites. **Critical scoping caveat:** the run path wraps the cluster in a `try/except` emitting `PipelinePhase.GRAPH` phase events (core.py:870-873); the resume path does not. The helper must contain *only* the four validators + stubs (zero event emission) or the resume path would start emitting GRAPH events it never emitted, changing the audit record. **Behavior-safe** if held to exactly that span and call order.
- **[core.py:862 / resume.py:151 / run_core.py:180 / preflight.py:145] (LOW) `sink._on_write_failure` private-attribute access duplicated across multiple modules** *(merge: confirmed finding + needs-context item).* Correction to both: `_on_write_failure` is a *formally declared* `SinkProtocol` member (plugin_protocols.py:719), so a rename *is* mypy-caught at every site — the "no safety net" framing is overstated. What genuinely lacks one audited home is the duplicated `SimpleNamespace` stub-shaping (4 sites). The underscore values, *including None*, are surfaced deliberately: `validate_sink_failsink_destinations` keys a framework-bug error on None. Any accessor must return the raw injected value unchanged. **Recommendation:** mild — extract the stub construction into a single `validation.py` helper; leave the protocol attribute as-is. Adjudicate against release-freeze policy.
- **[resume.py:533, 902 / core.py:126-133] (LOW) The core↔resume import cycle is a parking-spot artifact, not an inherent dependency.** `_RunFailedWithPartialResultError` is an 8-line exception with no `Orchestrator` dependency, currently forcing two deferred imports. **Fix:** move it to the leaf `types.py`. Corrections: only resume.py:902 is fully eliminated (533 still imports `prepare_for_run`, which legitimately stays); and an external importer (`test_t18_characterization.py:35` imports from `core`) requires keeping a re-export in core.py or updating the test. **Behavior-safe** with the re-export.
- **[resume.py:377-386] (LOW) `ResumeCoordinator` holds both `checkpoint_manager` and `checkpoints` (which wraps the same manager) — dual custody.** Raw manager used only to build `RecoveryManager` (436). Correction: the "duplicated None-handling" claim is wrong — the two guards have *different* policies (resume raises `OrchestrationInvariantError`; coordinator silently returns). **Fix:** either add a `make_recovery_manager`/accessor on `CheckpointCoordinator` (only safe if the resume site keeps raising on None), or just document why resume needs direct access. **Behavior-safe:** the documentation option is safe; the accessor option is not unconditionally safe (could alter the raise-vs-return contract).
- **[source_iteration.py:469-481] (LOW) `source_span` closes before generator rows are consumed** — the OTel span covers only `load()` + first-row probe, under-reporting streaming sources. Corrections: eager sources are fully covered, and crucially this is *not* the audit record (the audit attribution is `track_operation(source_load)`, which correctly wraps the whole loop) — observability fidelity only. **Fix:** document the limitation, or rename to `source_load_span`. **Behavior-safe:** Yes.
- **[resume.py:787-793] (LOW) Generic resume failure ceremony emits all-zero FAILED metrics**, discarding partial counters from re-drive, while the `_RunFailedWithPartialResultError` arm forwards them. Correction: RunSummary is *not* the audit legal record (it's a CLI/observability event; `finalize_run(FAILED)` IS written) — this is an event-surface inconsistency, not legal-record corruption, hence LOW. The suggested fix (re-derive counters in the except) is also fragile (the most likely trigger is a fault in that very derive call; `audit_counters` may be unbound). **Behavior-safe:** No; current conservative all-zero fallback is defensible. Recommend no auto-fix.

## 4. Type-Design Improvements

- **[types.py:60-103] (HIGH) `RowProcessorHandle` protocol erases all signatures on the audit-critical processor surface.** Six of eleven methods are `(self, *args: Any, **kwargs: Any) -> Any` — the central per-row contract the orchestrator drives every row through, with zero static checking. `process_row` returns `Any`, consumed untyped at source_iteration.py:616-617. `get_aggregation_buffer_count` keeps `-> int`, proving the pattern is incidental. **Fix (corrected):** give the high-traffic methods real signatures — but the finding's sketch is *wrong*: it marks params keyword-only (`*,`) when they are positional-or-keyword, omits `coalesce_node_id`/`coalesce_name`, and drops `process_token`'s required `current_node_id`. **Derive signatures verbatim from `processor.py` (1857, 1933, 2012).** `SourceRow` is the correct `source_row` type. **Behavior-safe:** annotation-only (no runtime change), but the *specific* signatures must be corrected before applying.
- **[run_core.py:406] (LOW) `initialize_run_context` accepts `Mapping[str, ...]` but re-keys to `NodeID` before forwarding.** Purely a static-typing gap (`NodeID = NewType(...)`, runtime no-op). Correction: the true source is `ResumePoint.node_id: str` (checkpoint.py:45); fixing only run_core.py:406 + types.py:516 will *not* let you delete the re-keying comprehension — you'd also need to retype the contract dataclass and resume.py:423. **Behavior-safe:** Yes (annotation-only).
- **[types.py:457-489] (LOW) `LoopContext` encodes its mutable/read-only invariant only in a comment.** Three de-facto field categories, identical dataclass mutability; nothing prevents reassigning `loop_ctx.config`/`processor` mid-run. No live violator (grep clean). Correction: the docstring is *already* inaccurate — `last_token_id` sits below the "never reassigned" separator but IS reassigned (source_iteration.py:577/617). **Fix:** first correct the docstring (cheapest, honest). A `__setattr__` guard is risky on a `slots=True` dataclass with legitimate `last_token_id` writes; the frozen-sub-bundle split is a real refactor, not a quick win. **Behavior-safe:** No for the structural options.
- **[ceremony.py:98] (LOW) `pending_exc = sys.exc_info()[0]` holds the exception *type*, not the instance** — the name implies an instance. Truthiness gate is correct either way. **Fix:** rename to `pending_exc_type` (preferred over switching to `[1] is not None`). **Behavior-safe:** Yes.

## 6. Test-Coverage Gaps

Mined from the verdict notes (no finding was categorized "test", but the adversarial checks surfaced concrete gaps):

- **cleanup-error AND pending-exception interaction is unpinned.** `test_orchestrator_cleanup.py` / `test_resume_failure.py` only exercise the *no-pending-exception* branch; the masking behavior in cleanup.py:120-124 (the HIGH bug) is latent/unspecified. Add a test asserting that a cleanup-hook failure during a partial-result run preserves the real counters and original error.
- **`rows_diverted` omitted from `accumulate_flush_result` assertions** (types.py:184-220). `test_accumulate_adds_to_existing` asserts 9 scalars but skips `rows_diverted`, and the `__add__`/field tests stop at `rows_buffered` — so the most-recently-added counter being dropped would go uncaught (the exact silent-undercount failure mode). Close this gap *before* any tuple-driven dedup refactor.

- **`safe_flush_telemetry` has no direct test reference** — no test pins its exception contract, so the MEDIUM broadening fix has no regression guard. Add one (including the non-`TelemetryExporterError` masking case).
- **Empty-branches coalesce shutdown-checkpoint outcome is unpinned** (checkpointing.py:204-216) — add a test that pins whether a checkpoint is written for a pending entry with empty `branches` + non-empty `lost_branches`.

## 7. Needs-Context / Open Questions

For the maintainer to adjudicate — these depend on caller behavior or policy not visible in the package:

- **[checkpointing.py:185-190] Coalesce branch produces a semantically mismatched `(token_id, node_id)` anchor when all branches are empty.** Mechanism confirmed (top-level anchor records a source-derived `last_token_id` against a coalesce node). **But the claimed recovery mis-attribution is refuted:** `get_unprocessed_row_data` keys off `sequence_number` and the state objects' own token lists, never the top-level anchor; `resume_point.token_id` is never read. Net: an audit-row cosmetic blemish, no recovery harm in `src/elspeth/`. **Open question:** does any *external* consumer treat `Checkpoint.node_id/token_id` as token-node provenance for audit reporting? None found in-tree. The finding's suggested patch is on the wrong branch (`elif`) and would flip a created checkpoint into a *skipped* one in the degenerate case — not safe.
- **[source_iteration.py:576-577] `last_token_id` advanced for quarantine rows only when the sink bucket is non-empty.** The guard's false branch is unreachable for a well-formed quarantine (`handle_quarantine_row` always appends one pair; sink validated non-None). **The finding's stale-checkpoint consequence is refuted** — checkpointing.py:192-197 scans `pending_tokens` first, so the just-appended token resolves the anchor; the `last_token_id` fallback never fires here. Net: a redundant defensive guard, cosmetic. **Open question:** worth replacing the redundant guard with an assert (surfaces the "impossible" per the finding's own goal) — but that adds a crash path, so not behavior-safe on the audit path. Recommend leaving as-is.
