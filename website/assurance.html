Assurance is the product

Validation and audit are core properties.

The Landscape database is the canonical evidentiary record — source rows, node states, external calls, payload hashes, route decisions, terminal outcomes, and artifact provenance. Telemetry and logs are secondary.

Assurance properties

Landscape audit trail

Complete row lineage with content hashes. Export signed (HMAC-SHA256) with a running hash for chain-of-custody review.

Two-axis terminal model

Lifecycle outcome and path/provenance are separated — "did it succeed?" and "how did it get there?" are both explicit.

Declaration-trust + VAL

Plugin declarations the graph builder trusts are backed by runtime VAL manifests, invariant tests, and CI scanners.

Secret discipline

Secrets resolve at execution boundaries, are fingerprinted for audit, and are never persisted as raw values.

Strict response schemas

Web execution responses forbid unknown fields, so internal drift fails loudly instead of being silently coerced.

Terminal statuses

completed · completed_with_failures · failed · empty · cancelled — run accounting never collapses to one number.

Show, don't tell

One row's defensible trail.

This is real explain output from a bundled example run — not a mock-up. It reconstructs why a single row reached the approved sink: the source fingerprint it carried, the gate it passed, and the routing decision that sent it there.

elspeth · explain one row

Outcome and path are separate axes

The two-axis terminal model in practice: SUCCESS answers did it succeed?; GATE_ROUTED answers how did it get there? "It passed" and "a gate sent it to approved" are recorded as distinct facts.

Every decision is bound to its evidence

The source row carries a content hash; the routing event carries a reason_hash that ties the destination to the exact condition that produced it. Nothing in the path is asserted without a fingerprint.

Reproducible, and exportable as evidence

The same records export signed (HMAC-SHA256) with a running hash for chain-of-custody review — the trail a regulated or audited assessment needs to reconstruct a single decision after the fact.

Node ids, source-data and reason hashes are abbreviated here (marked …) for display; the run records them in full. Reproduce it against any bundled run under examples/*/runs/audit.db.

Data trust model

Three tiers, one rule: coerce only at the boundary.

Once data enters the pipeline with valid types, downstream transforms trust those types. Wrong types downstream are upstream bugs to fix — not data-quality issues to handle gracefully.

Tier 1

Engine-owned records

Audit database, checkpoints, runtime records. A corrupted audit record means tampering — crash immediately.

Full trust · crash on anomaly
Tier 2

Validated pipeline data

Rows after source validation. Types are valid; wrap operations where row content can still fail.

Elevated trust
Tier 3

External input

Files, API/LLM responses, source data. Validate at the boundary, coerce where allowed, quarantine malformed rows.

Zero trust · quarantine
