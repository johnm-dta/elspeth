Two authoring surfaces, one engine

Hand-edit YAML, or build by conversation.

Both surfaces target the same primitives, plugin contracts, graph validation, and Landscape audit trail. The Web Composer is an authoring surface over the substrate — not an alternative engine.

Authoring surfaces

YAML operator path

For operators in sensitive, regulated, or defence-adjacent workflows. The pipeline can be read, reviewed, versioned, and explained before it ever runs.

  • Declarative DAG — every edge explicitly named and validated
  • Hierarchical settings with profile overrides
  • validate, run, explain, resume from the CLI

Web Composer

For knowledge workers building document QA, classification, routing, or review workflows. Since 0.7.0, Guided Composer has been LLM-primary and staged: it makes the graph, validation, and review visible before a pipeline is wired. Guided and freeform authoring use different interactions, but both commit through the same canonical proposal + validation path.

  • Guided — default-on for a first pipeline: a stepped walk through source → sink → transforms → wiring, prompting one stage at a time
  • Freeform — one open conversation for a fluent operator who drives the whole pipeline at once, no rails
  • Visible graph, preview validation, review, spec, YAML, and repair hints
  • Blob-backed sources and wired {secret_ref: NAME} references
  • Background runs streamed over WebSocket, safely cancellable

Current guided-staged limits: it cannot author cross-sink write-failure fallback or require-all coalesce shapes. Use freeform and guided-full for those pipelines.

Propose, validate, then wire

Make a durable change without collapsing review.

In 0.7.1, proposal and candidate handling is durable: a candidate remains separate until review and wiring are confirmed. This preserves a clear path from an authoring suggestion to the validated pipeline it changes.

Components for the graph

Across the authoring paths, proposals can include plural sources, transforms, sinks, gates, structural queues, fork, and coalesce nodes.

Text in and out

Use a text source to introduce text into a pipeline and a text sink to write a text result.

Structural queue

A structural queue fans multiple upstream routes into one downstream node without merging rows or schemas. It is not a human review queue or a message broker.

Parallel lineage

Fork into branches — without losing the thread.

A gate can fork one row into parallel branches that traverse the DAG independently, then a coalesce node joins them back. The audit trail follows every step, so a branched decision is still fully reconstructable.

Execution model

Concurrent, but still deterministic.

LLM calls are the slow part of most pipelines, so an LLM transform fans its per-row requests out across a bounded thread pool — then puts the results back in order. Throughput comes from overlapping I/O-bound model calls inside a transform, not from parallelising the graph, and never at the cost of a reproducible run.

Bounded fan-out

Per-row LLM and API calls run through a semaphore-bounded ThreadPoolExecutor, so a single LLM transform overlaps in-flight requests up to a fixed ceiling instead of waiting one row at a time.

AIMD back-pressure

Concurrency adapts to the provider: additive-increase while calls succeed, multiplicative-decrease on capacity errors — the pool backs off under load instead of hammering a rate-limited endpoint.

Reorder buffer

Concurrent results arrive out of order, then a reorder buffer re-sequences them back to submission order before they flow downstream. Parallel execution never reshuffles your rows.

Frozen row contexts

Rows crossing a thread boundary are exposed as read-only views (MappingProxyType), so concurrent workers share no mutable state and cannot race each other.

Contained regex, force-killable

The only OS-level subprocess is a single-worker, force-killable process that runs regex in isolation, so a catastrophic-backtracking (ReDoS) pattern can be killed rather than hang the run. Containment for safety — not parallelism for speed.

Preserves the deterministic audit

Fan-out lives inside the transform layer; the orchestrator and Landscape audit stay single-writer and ordered. The reorder buffer is what lets concurrency keep the audit reproducible, not undermine it.
