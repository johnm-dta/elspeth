Reviewable by design

From install to a validated run.

Choose the CLI for a reviewable source checkout, or set up the Web Composer for a local, staged authoring experience. Both validate before they run.

Install and run from the CLI

1

Install from a current source checkout

Python 3.12+. Clone the repository and use an editable install with uv. This tracks your current source checkout; it is not an immutable 0.7.1 artifact.

$ git clone https://github.com/johnm-dta/elspeth.git && cd elspeth
$ uv venv && source .venv/bin/activate
$ uv pip install -e ".[dev]"
2

Validate a tracked example

Start with the repository's examples/threshold_gate/settings.yaml. Preflight checks graph structure, route targets, schema compatibility, and secret references.

$ elspeth validate --settings examples/threshold_gate/settings.yaml
3

Run & explain

Execute the example, then explore why any row reached its destination.

$ elspeth run --settings examples/threshold_gate/settings.yaml --execute
$ elspeth explain --run <run_id> --row <row_id> --database examples/threshold_gate/runs/audit.db --no-tui
Outcome:   SUCCESS
Path:      GATE_ROUTED
Sink:      output
Important upgrade posture

Pre-1.0 databases are recreated, not migrated.

APIs, interfaces, and schemas are unstable before 1.0; ELSPETH makes no backward-compatibility promise. From 0.7.0 to 0.7.1, SESSION_SCHEMA_EPOCH changes 26 → 35, the guided schema changes 7 → 10, and Landscape SQLITE_SCHEMA_EPOCH changes 22 → 29.

Archive or export needed evidence, stop the old service, and recreate incompatible session and Landscape stores. Do not run older code over recreated databases: there is no in-place migration. The separate auth.db is not the session or Landscape store.

Web Composer setup

Build the local Web Composer.

The Web Composer is a separate local setup path. Build its frontend, configure a secret, create a user, then start the web service.

Install, build, and run

Start from a fresh source checkout. Node.js 24 and npm 11 are required before building the frontend; then set a local secret before creating the first user.

$ git clone https://github.com/johnm-dta/elspeth.git && cd elspeth
$ uv venv && source .venv/bin/activate
$ uv pip install -e ".[webui,dev]"
$ cd src/elspeth/web/frontend
$ npm install
$ npm run build
$ cd ../../../..
$ export ELSPETH_WEB__SECRET_KEY="$(openssl rand -base64 32)"
$ elspeth composer users add demo --password "choose-a-strong-password" --display-name "Demo User" --email "demo@example.com"
$ elspeth web --host 127.0.0.1 --port 8451
Supported deployment profile

Operate one web task on AWS ECS.

The supported profile is one ECS Fargate web task with Aurora PostgreSQL, EFS, task-role S3 and Bedrock access, Cognito/OIDC, and CloudWatch/X-Ray. Deployments use planned downtime and zero overlap.

Follow the docs/runbooks/aws-ecs-deployment.md runbook for the current operational contract. The current code and runbook are the source of truth.
