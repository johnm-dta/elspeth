Same framework, different plugins

Built for high-stakes, accountable processing.

Wherever a decision needs to be explainable to an auditor, ELSPETH maps the work onto Sense → Decide → Act and keeps a full audit trail behind it.

Patterns, not packaged solutions

Built in

Built-in sources read CSV, JSON, text, Azure Blob, and Dataverse data. Built-in destinations write CSV, JSON, text and other local or object-store files, plus Azure Blob, Dataverse, database, and Chroma targets.

Extend with custom plugins

API feeds, message queues, alerts, tickets, and domain-specific ML models require custom plugins and the operator's own service configuration.

When to reach for it

A good fit when "why did it do that?" matters.

Explainable to auditors

Decisions with regulatory or compliance requirements, or legal accountability for the outcome.

Mixed automated + human review

Workflows that route some rows to people and need an evidentiary record of which went where, and why.

High-stakes processing

Where the wrong output causes operational, legal, safety, financial, or security harm.

Honest about fit

When to reach for something else.

The audit machinery that makes ELSPETH defensible is overhead you don't want for throughput-first work. If you don't need the trail, a leaner tool will serve you better.

When to choose a different tool
If you need…Consider insteadWhy
High-throughput ETLSpark · dbtBulk transformation at scale, where per-row lineage and audit evidence aren't the point.
Sub-second streamingFlink · Kafka StreamsLow-latency event processing. ELSPETH is not a sub-second streaming engine; its orchestrator prioritises deterministic, auditable work.
Simple scripts, no auditPlain PythonOne-off or low-stakes work where a reviewable trail would be ceremony, not value.

ELSPETH trades raw throughput for evidence. That's the right trade only when "why did it do that?" has to have an answer.
